# Springプロジェクト向け 三層分離型テスト基盤設計書

## 1. 文書情報

- 文書名: Springプロジェクト向け 三層分離型テスト基盤設計書
- 提案名: 三層分離型テスト基盤構成
- 目的: 開発中のSpringプロジェクトに対して、保守性・実行効率・CI適合性を両立したテスト基盤を導入するための標準設計を定義する
- 適用対象: Maven採用のSpring / Spring Boot プロジェクト
- 想定利用者:
  - 開発者
  - テックリード
  - 品質保証担当
  - CI/CD運用担当
- 本文書の位置付け: 実装前の設計仕様書。Codex実装指示書、およびレビュー基準の上位文書として扱う

---

## 2. 背景

Springフレームワークを採用した開発プロジェクトでは、以下のような課題が発生しやすい。

- テストコードの書き方が開発者ごとにばらつく
- Spring起動を伴う重いテストが増え、ローカル実行やCIの速度が低下する
- 単体テスト、結合テスト、統合テストの境界が曖昧になる
- IDE実行に依存し、CI上の標準確認手順が弱くなる
- `mvn test` と `mvn verify` の責務分担が未整理のまま運用される

これを解消するため、本設計では以下を基本方針とする。

- テスト記述の標準化
- テスト実行経路の標準化
- Spring連携テストの適用範囲の明確化
- 重い統合テストの分離
- CI/CDへの自然な接続

---

## 3. 設計方針

### 3.1 基本方針

本プロジェクトでは、テスト基盤を以下の三層に分離して設計する。

1. テスト記述層
2. テスト実行統制層
3. Spring統合検証層

各層の責務を分離することで、以下を実現する。

- テストコードの可読性向上
- 実行速度の最適化
- 保守性向上
- CI/CD組み込みの容易化
- テスト失敗時の原因切り分け容易化

### 3.2 採用方針

本プロジェクトでは、以下を採用する。

- JUnit 5
- Maven標準テスト実行 (`mvn test`)
- Maven検証フェーズ (`mvn verify`)
- Spring Test / Spring Boot Test
- Maven Surefire Plugin
- Maven Failsafe Plugin
- 必要に応じて Mockito
- 必要に応じて AssertJ
- 必要に応じて JaCoCo

### 3.3 非採用方針

以下は原則として採用しない、または制限して採用する。

- すべてのテストを `@SpringBootTest` ベースで記述する方式
- 単体テストと統合テストの命名・実行経路を混在させる方式
- IDE実行のみを前提とした非標準運用
- `mvn test` の対象に重い統合テストを無差別に含める方式

## 4. 用語定義

### 4.1 JUnit
Java向けテストフレームワーク。テストコードの記述、アサーション、パラメータ化テスト、例外検証などを担う。

### 4.2 `mvn test`
Mavenの test フェーズを実行する標準コマンド。主として単体テストおよび軽量テストの実行入口とする。

### 4.3 `mvn verify`
Mavenの verify フェーズを実行する標準コマンド。統合テストを含む総合検証入口とする。

### 4.4 Spring Test / Spring Boot Test
Springコンテキスト、Bean解決、MVC、JPAなどSpring固有要素のテストを支援する機構。

### 4.5 Surefire
Mavenの通常テスト実行プラグイン。`test` フェーズで主に単体テストを実行する。

### 4.6 Failsafe
Mavenの統合テスト実行プラグイン。`integration-test` および `verify` フェーズで統合テストを実行する。

---

## 5. 責務分担

### 5.1 JUnitの責務
JUnitはテスト記述の標準フレームワークとする。

責務:
- テストケース記述
- アサーション実装
- 例外検証
- パラメータ化テスト
- 単体テスト構造化
- テストコードの可読性担保

JUnitは「何をどう検証するか」をコード化する責務を持つ。

### 5.2 `mvn test` の責務
`mvn test` は日常開発およびCI前段における標準実行入口とする。

責務:
- 単体テスト一括実行
- 軽量なスライステスト一括実行
- 開発者ローカルでの標準確認
- CIの一次品質ゲート
- 失敗時のビルド停止

`mvn test` は「日常的な品質確認の標準運用入口」とする。

### 5.3 `mvn verify` の責務
`mvn verify` は統合テストを含む総合検証入口とする。

責務:
- 統合テスト実行
- 重いSpring起動テストの実行
- DB接続・外部連携を含む検証
- リリース前の総合品質確認

`mvn verify` は「重い検証を含む後段品質ゲート」とする。

### 5.4 Spring Test / Spring Boot Test の責務
Spring Test はSpring固有要素を検証するために限定的に利用する。

責務:
- Bean構成確認
- DI成立確認
- Controller層のHTTP入出力確認
- Repository/JPA動作確認
- 設定値読込確認
- アプリケーション起動整合性確認

## 6. テスト分類設計

### 6.1 単体テスト
対象:
- Serviceロジック
- Domainロジック
- Utilityクラス
- 純粋関数的処理
- 入出力変換ロジック

手段:
- JUnit
- Mockito
- AssertJ

原則:
- Springコンテキストを起動しない
- 高速に実行できることを重視する
- 外部依存はモック化する
- テスト件数の主力は単体テストとする

### 6.2 スライステスト
対象:
- Controller層
- Repository層
- JSON変換
- バリデーション
- 永続化マッピング

手段:
- `@WebMvcTest`
- `@DataJpaTest`
- 必要に応じて `@JsonTest`
- 必要に応じて `@RestClientTest`

原則:
- 必要な部分のみSpringを起動する
- フルコンテキスト起動を避ける
- 単体テストでは検証しにくいSpring連携を補う

### 6.3 統合テスト
対象:
- アプリケーション全体の起動確認
- コンポーネント連携
- 複数レイヤをまたぐ業務シナリオ
- 設定ファイル・Bean構成・トランザクションを含む総合検証

手段:
- `@SpringBootTest`
- 必要に応じて Testcontainers
- 必要に応じて実DBまたはテストDB

原則:
- 件数を絞る
- 重い処理は `mvn verify` 側へ分離する
- 単体テストの代替にしない

---

## 7. 実行方針

### 7.1 `mvn test` 実行対象
`mvn test` では以下を対象とする。

- 単体テスト
- 軽量スライステスト

目的:
- 高頻度に実行可能な速度を維持する
- 開発者の手元確認を簡潔にする
- CI一次チェックを短時間で完了させる

### 7.2 `mvn verify` 実行対象
`mvn verify` では以下を対象とする。

- `mvn test` 対象一式
- 統合テスト
- 重いSpring起動テスト
- 必要な外部接続を伴う検証

目的:
- リリース前またはCI後段で総合品質を担保する
- 実運用近い環境で破綻がないことを確認する

### 7.3 IDE実行の扱い
IDEからの個別実行は許可するが、標準運用とはしない。

方針:
- 開発中の局所確認にはIDE実行を活用してよい
- ただし、レビュー前またはコミット前の標準確認は `mvn test` を基準とする
- CIは `mvn test` または `mvn verify` を必須とする

---

## 8. 命名規約

### 8.1 単体テスト
- 命名規約: `*Test.java`

例:
- `UserServiceTest`
- `PriceCalculatorTest`

用途:
- 単体テスト
- 軽量スライステスト

### 8.2 統合テスト
- 命名規約: `*IT.java`

例:
- `UserRepositoryIT`
- `OrderFlowIT`

用途:
- 統合テスト
- フルSpringコンテキスト依存テスト
- DBまたは外部依存を含む重い検証

### 8.3 エンドツーエンド相当テスト
必要な場合のみ以下を許容する。

- 命名規約: `*E2E.java`

ただし、本設計の主対象はMaven標準テスト基盤であり、E2Eは別基盤とする可能性があるため、採用時は別途規約化する。

---

## 9. ディレクトリ方針

### 9.1 基本方針
テストコードは原則として以下に配置する。

- `src/test/java`
- `src/test/resources`

### 9.2 配置方針
本設計では、物理ディレクトリよりも命名規約と実行フェーズで分類する。

- `*Test.java` は Surefire 対象
- `*IT.java` は Failsafe 対象

必要に応じて以下のような論理分離は許容する。

- `...service...`
- `...controller...`
- `...repository...`
- `...integration...`

ただし、最終分類は命名規約を優先する。

## 10. Springテスト適用境界

### 10.1 `@SpringBootTest`
適用対象:
- 全体起動確認
- 複数Bean連携確認
- フレームワーク全体挙動の検証
- 本番構成に近い統合確認

禁止または抑制方針:
- 単純なServiceロジック検証に使わない
- モックで十分なテストに使わない
- テスト件数の主力にしない

### 10.2 `@WebMvcTest`
適用対象:
- Controller単体近傍のHTTP入出力検証
- バリデーション検証
- ステータスコード検証
- JSONレスポンス検証

### 10.3 `@DataJpaTest`
適用対象:
- Repositoryの永続化動作確認
- JPAマッピング確認
- クエリ動作確認

### 10.4 Mockito
適用対象:
- Serviceロジック
- 外部依存分離
- コラボレータ動作の疑似化

方針:
- 単体テストではMockitoを優先活用する
- Spring起動よりもモック利用を優先する

---

## 11. Maven設定方針

### 11.1 依存関係
基本依存は `spring-boot-starter-test` を採用する。

含意:
- JUnit 5
- Spring Test
- Mockito
- AssertJ
- Hamcrest 等を一括利用可能とする

必要に応じて追加:
- JaCoCo
- Testcontainers
- DBテスト用依存
- WireMock等の外部連携テスト補助ライブラリ

### 11.2 Surefire設定方針
Surefireは `test` フェーズで以下を実行する。

対象:
- `*Test.java`

除外:
- `*IT.java`
- `*E2E.java`（採用時）

### 11.3 Failsafe設定方針
Failsafeは `integration-test` および `verify` フェーズで以下を実行する。

対象:
- `*IT.java`

目的:
- 重い統合テストを日常テストから分離する
- CI後段またはリリース前検証に適用する

---

## 12. CI/CD運用方針

### 12.1 ローカル開発
開発者の標準確認手順:
1. 変更箇所に対する対象テストをIDEまたは限定実行で確認
2. コミット前に `mvn test` を実行
3. 必要に応じて `mvn verify` を実行

### 12.2 CI前段
CI前段では以下を実行する。

- `mvn test`

目的:
- 高速フィードバック
- 破壊的変更の早期検知
- PR段階の品質担保

### 12.3 CI後段またはリリース前
CI後段またはリリース前では以下を実行する。

- `mvn verify`

目的:
- 統合的整合性の検証
- 実運用近似条件での破綻検知
- リリース判定材料の取得

---

## 13. 品質基準

### 13.1 単体テスト品質基準
- Spring起動不要なテストはSpringを起動しないこと
- 失敗時に原因が即座に追跡可能であること
- テスト名が仕様を表していること
- 1テストで多責務を持たないこと

### 13.2 スライステスト品質基準
- 対象レイヤが明確であること
- 必要以上のBeanをロードしないこと
- Controller / Repository など責務境界が保たれていること

### 13.3 統合テスト品質基準
- 全体結合性または起動整合性を検証していること
- 単体テストで代替できる内容を重複していないこと
- 実行時間に見合う検証価値があること

---

## 14. 導入手順方針

### 14.1 導入前確認項目
導入前に以下を確認する。

- 既存 `pom.xml` の依存関係
- 既存 Surefire / Failsafe 設定有無
- 既存テスト命名規約
- 既存CI設定
- 既存テストの実行時間
- `@SpringBootTest` の利用実態
- DB依存テストの有無
- 外部API依存テストの有無

### 14.2 導入ステップ
1. テスト依存の棚卸し
2. 命名規約の標準化
3. Surefire / Failsafe の責務分離
4. `mvn test` / `mvn verify` 実行設計の確定
5. 既存テストの分類見直し
6. 重い `@SpringBootTest` の整理
7. CIジョブへの反映
8. レビュー観点の整備

### 14.3 段階導入方針
既存プロジェクトでは一括全面改修よりも段階導入を推奨する。

推奨順:
1. 新規追加テストから命名規約を適用
2. 重い既存テストを `*IT` に再分類
3. `mvn verify` 側へ順次分離
4. 単体テストのSpring依存除去を継続的に実施

---

## 15. 禁止事項

以下を原則禁止とする。

- 単体テスト相当の検証を `@SpringBootTest` で書くこと
- 統合テストを `*Test` 命名で `mvn test` 側へ混在させること
- テスト実行基準を開発者ごとに変えること
- Maven標準実行を通さずCI品質判定を行うこと
- 外部接続を伴う重いテストを無差別に前段CIへ含めること

## 17. リスクと対策

### 17.1 リスク
- 既存テスト資産との不整合
- 命名規約変更に伴う一時的混乱
- CI設定変更による初期不安定化
- 重いテストの分類作業負荷
- 開発者が `@SpringBootTest` に依存した既存習慣を持つ場合の移行負荷

### 17.2 対策
- 段階導入を採用する
- 命名規約と実行方針をレビュー基準に組み込む
- 初回導入時は代表モジュールで先行適用する
- CI前段／後段の責務を明文化する
- サンプルテストを整備し、実装パターンを統一する

## 18. 完了条件

本設計の適用完了条件は以下とする。

1. `pom.xml` 上でテスト依存関係が標準化されていること
2. Surefire と Failsafe の役割分担が明確化されていること
3. `*Test` / `*IT` 命名規約が定義され、適用されていること
4. `mvn test` で単体テスト＋軽量スライステストが実行されること
5. `mvn verify` で統合テストが実行されること
6. `@SpringBootTest` の利用範囲が統合テスト中心に制御されていること
7. CIに `mvn test` および必要に応じて `mvn verify` が組み込まれていること
8. 開発者向け運用ルールが共有されていること

## 19. 最終結論

本設計では、Springプロジェクトに対し、JUnit と Maven標準実行、およびSpring Testを競合要素としてではなく、責務分離された補完要素として扱う。

最終構成は以下の通りとする。

- JUnit: テスト記述の標準
- `mvn test`: 日常的な品質確認の標準入口
- `mvn verify`: 統合検証の標準入口
- Spring Test / Spring Boot Test: Spring固有挙動の検証手段
- Surefire / Failsafe: 実行責務分離の基盤

これにより、本プロジェクトでは「速く回る単体テスト」と「必要最小限のSpring連携テスト」と「重い統合テスト」を分離し、保守性・実行効率・品質保証のバランスを取るテスト基盤を実現する。

## 20. 付録: 運用上の推奨ルール

### 20.1 開発者ルール
- ロジック検証はまず単体テストで考える
- Spring機能が必要な場合のみSpring Testを使う
- 全体起動テストは最後の手段とする
- PR前の基本確認は `mvn test` を基準とする

### 20.2 レビュールール
レビュー時は以下を確認する。

- このテストはSpring起動が本当に必要か
- `*Test` と `*IT` の分類は適切か
- `mvn test` の対象として妥当か
- 単体テストで代替可能なものを統合テストにしていないか
- テスト名が仕様を表現しているか

### 20.3 将来拡張
将来的には以下の拡張を許容する。

- カバレッジ計測導入
- Testcontainers導入
- 契約テスト導入
- APIテスト自動化
- 品質ゲートの閾値管理
- モジュール単位のテスト並列化
