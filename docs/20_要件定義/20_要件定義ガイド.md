# 20_要件定義ガイド

---

## 1. 文書の目的

本ガイドは、`docs/20_要件定義/` 配下に配置された要件・方針・補助仕様文書について、  
**どの文書を、どの目的で、どの順番で読むべきか** を明確にすることを目的とする。

現状の `20_要件定義` 配下には、開発の進行に応じて逐次追加された文書が複数存在している。  
これらはすべて不要な重複資料ではなく、それぞれ異なる判断軸や論点を担っている一方、  
**ファイルが横並びで置かれているだけでは、初見利用者にとって「何が正本で、何が補助仕様で、どこから読むべきか」が分かりにくい** 状態になりやすい。

そのため本ガイドでは、以下を明確化する。

- `20_要件定義` 配下の文書の位置づけ
- 現行正本と現行補助仕様の区別
- 利用目的別の推奨参照先
- 新規参加者や実装担当者向けの推奨読書順
- 今後の文書追加・統合・アーカイブ時の判断基準

## 位置づけ補足

`20_要件定義/` は、Q-Scout for Spring の現行要件、運用前提、主要方針を定義するためのフォルダです。
システム全体の骨格、責務分離、主要処理フローを正式に定義する文書は `30_基本設計/基本設計.md` とし、
本フォルダ内の設計寄り文書は、要件理解を補助する資料として扱います。

---

## 2. このフォルダの位置づけ

`docs/20_要件定義/` は、Q-Scout for Spring に関する  
**現行の要件、方針、制約、実装前提、補助仕様** を保持するためのフォルダである。

本フォルダは、プロジェクト全体の文書体系の中で、概ね以下の役割を担う。

- `docs/10_企画/`  
  プロジェクトの企画、背景、目的、提供価値、方向性を定義する文書群

- `docs/20_要件定義/`  
  現行実装・現行運用に関する要件、方針、制約、補助仕様を定義する文書群

- `docs/40_詳細設計/`  
  個別 UI や差分設計など、詳細な設計レベルの文書群

- `docs/90_アーカイブ/`  
  履歴資料、初期構想、旧フェーズ文書など、現行正本ではないが経緯保持のため残す文書群

したがって、`20_要件定義` 配下の文書は、  
**日常的な仕様判断や docs 整理判断において最も参照頻度が高い階層** と位置づける。

## 3. 最初に読むべき文書

### 3.1 初回参照時の基本方針

このフォルダの文書は、原則として **すべてを順番に読む前提ではない**。  
まずは **中核となる現行正本** を読んで全体像を把握し、その後、必要な論点に応じて補助仕様へ進む。

### 3.2 最低限読むべき文書

初めて Q-Scout for Spring の要件群を読む場合、まず以下を推奨する。

1. `Q-Scout-for-Spring-Web化フェーズ-Web-MVP-最小要件定義書-v1.0.md`  
2. `Q-Scout-for-Spring-Web化フェーズ-全体設計方針書-v1.0.md`  
3. `Web実行制約・運用前提-v0.1.md`  
4. `Q-Scout-for-Spring_多言語化対応仕様書_v1.0.md`

この4本で、概ね以下が把握できる。

- 何を Web MVP の成立条件とするか
- どのような構造で Web 化を進めたか
- 実行上どのような制約があるか
- 多言語化の適用範囲と運用方針は何か

### 3.3 追加で読むべき文書

以下は必要時に追加参照する。

- 既存 CLI 資産をどう扱ったか確認したい  
  → `Q-Scout-for-Spring-Web化フェーズ-既存CLI資産からの移行・流用方針-v1.0.md`

- Web 層の主要骨格や責務分離を把握したい  
  → `Q-Scout-for-Spring-Web化フェーズ-最小Webアーキテクチャ設計書-v1.0.md`

- 特定ルールの解釈や見直し方針を確認したい  
  → `ControllerToRepositoryDirectAccessRule_見直し方針書.md`

## 4. 現行正本

### 4.1 現行正本の定義

本ガイドにおける **現行正本** とは、  
**現行実装・現行運用の判断基準として優先的に参照すべき文書** を指す。

以下の特徴を持つ文書を正本とする。

- 現在のプロダクト状態と大きく矛盾していない
- 開発・docs整理・仕様判断の基準として使える
- 特定の補助論点ではなく、全体要件・全体方針・全体制約を扱っている

### 4.2 現行正本一覧

#### 1. `Q-Scout-for-Spring-Web化フェーズ-Web-MVP-最小要件定義書-v1.0.md`
**概要**  
Web MVP として何を成立条件とするかを定義した中核要件書。  
単画面、zipアップロード、同期解析、結果表示、レポート取得、エラー表示などの基本要求を定める。

**主な参照場面**
- Web MVP の最小成立条件を確認したい時
- UI / 入出力 / 実行フローの最小要件を確認したい時
- docs 整理時に「これは現行要件に含まれるか」を判断したい時

#### 2. `Q-Scout-for-Spring-Web化フェーズ-全体設計方針書-v1.0.md`
**概要**  
Web 化フェーズ全体の設計方針を示す文書。  
CLI資産活用、薄い Web 層、共通解析コア活用といった全体方針を把握するための基準文書。

**主な参照場面**
- Web 化の基本思想を確認したい時
- docs の方針整理において、全体設計観点から判断したい時
- 個別仕様の重みづけを判断したい時

#### 3. `Q-Scout-for-Spring_多言語化対応仕様書_v1.0.md`
**概要**  
日本語 / 英語対応の範囲、適用対象、実装・運用の方針を示す文書。  
UI や人間向けレポートの多言語化の基準となる。

**主な参照場面**
- i18n 対応範囲を確認したい時
- 文言変更や locale 対応の影響範囲を確認したい時
- docs 整理上、多言語化文書の位置づけを判断したい時

#### 4. `Web実行制約・運用前提-v0.1.md`
**概要**  
Web 実行時の運用制約、前提条件、実行環境制約を整理した文書。  
タイムアウト、サイズ制限、同期処理、ステートレス運用などを確認するための基準文書。

**主な参照場面**
- 実行制約や運用条件を確認したい時
- 要件変更が実行条件と矛盾しないか確認したい時
- docs 側で制約表現の整合を確認したい時

## 5. 現行補助仕様

### 5.1 現行補助仕様の定義

**現行補助仕様** とは、現行理解を強く補助するが、  
単体では全体要件の正本とは言いにくい文書を指す。

以下の特徴を持つ。

- 特定の論点や設計骨格を補足する
- 現行理解には有用だが、全体仕様判断の唯一の基準ではない
- 正本を読んだ後に、必要時に参照するのが適切である

### 5.2 現行補助仕様一覧

#### 1. `Q-Scout-for-Spring-Web化フェーズ-既存CLI資産からの移行・流用方針-v1.0.md`
**概要**  
既存 CLI 資産をどのように維持・流用し、どの責務を新設したかを示す方針書。  
CLI と Web の関係、共通解析サービスの位置づけ理解に有用である。

**主な参照場面**
- CLI 資産流用方針を確認したい時
- `SharedAnalysisService` 周辺の役割を理解したい時
- なぜ CLI を残しつつ Web 化したかを説明したい時

#### 2. `Q-Scout-for-Spring-Web化フェーズ-最小Webアーキテクチャ設計書-v1.0.md`
**概要**  
Web 層・サービス層・共通解析サービスの主要骨格を理解するための設計書。  
現行実装理解には有用だが、後続追加まで完全網羅した正本ではない。

**主な参照場面**
- Web 全体骨格を把握したい時
- コントローラ / サービス責務の全体像を知りたい時
- 実装差分確認時の補助資料として使いたい時

#### 3. `ControllerToRepositoryDirectAccessRule_見直し方針書.md`
**概要**  
特定ルールの解釈見直し方針を示す補助仕様。  
全体要件書ではなく、個別ルール理解を補強する文書である。

**主な参照場面**
- 当該ルールの重大度・評価方針を確認したい時
- ルール精度改善議論を行う時
- docs 上のルール説明と整合を取りたい時

## 8. 現時点の整理方針

`20_要件定義` 配下の文書は、現時点では **全面統合して1本化しない**。  
理由は以下のとおりである。

1. 各文書が異なる論点を担っている
2. すべてを1本へ統合すると巨大化し、更新しにくくなる
3. 現在は「何を読むべきか」が問題であり、「すべてが不要な重複」であるとは言えない
4. 段階的に整理した方が docs と実装の整合を保ちやすい

したがって現時点では、  
**中核正本＋補助仕様＋履歴資料** の三層構造を維持しつつ、  
本ガイドで読み順と参照目的を明確化する方針を採る。

---

## 9. 将来の統合方針

将来的に、要件群が十分安定し、文書の役割分担が固定化された場合には、  
**統合版要件定義書** を別途作成することは可能である。

ただし、その際も以下を満たすことを条件とする。

- 現行正本同士の重複が明確になっていること
- 統合後の文書が過大化しすぎないこと
- 補助仕様まで無理に統合しないこと
- 統合前文書は即削除せず、履歴または補助資料として扱いを定義すること

現時点では、統合よりも **索引・分類・読み順整理の方が優先度が高い**。

## 10. 更新ルール

本ガイドは、以下のいずれかが発生した場合に更新を検討する。

- `20_要件定義` 配下へ新規文書を追加した時
- 既存文書の位置づけが変わった時
- 正本 / 補助仕様 / 履歴資料 の分類を変更した時
- docs 全体構成を見直した時
- 新規参加者や利用者から「どれを読めばいいか分からない」という状態が再発した時

### 10.1 新規文書追加時の分類基準

**現行正本**
- 全体要件・全体方針・全体制約に関わる
- 仕様判断の基準として優先される

**現行補助仕様**
- 特定テーマや特定責務を補足する
- 正本理解を補助する

**履歴資料**
- 現行判断の基準ではない
- 経緯や当時の意思決定を残す目的で保持する

## 12. 最終結論

`docs/20_要件定義/` 配下の文書は、すべてを機械的に統合すべき段階にはまだない。  
現時点で最も重要なのは、**中核となる正本を明確にし、必要時に補助仕様へ進める状態を作ること** である。

したがって、本ガイドは以下を最終原則とする。

- すべての文書を最初から読む前提にしない
- 仕様判断は現行正本を優先する
- 補助仕様は必要時に参照する
- 履歴資料は現行正本と混同しない
- 新規文書追加時も、本ガイドを起点に分類と導線を維持する
