## 3. システム概要

### 3.1 プロダクト概要

Q-Scout for Spring は、Spring Boot / Maven プロジェクトを対象に、設計健全性、保守性および Spring ベストプラクティス準拠度を診断し、総合スコアおよび違反情報を出力する品質診断ツールである。

本ツールは単なる Lint ではなく、以下の特徴を持つ。

- ルールベースで設計上の問題候補を検出する
- 100点満点の減点方式でスコアを算出する
- 人間用 Markdown レポートを出力する
- AI投入用 Markdown を出力する
- Web UI から利用可能である
- CLI 実行経路も維持する

### 3.2 提供形態の全体像

現行の Q-Scout for Spring は、以下の提供形態を両立する。

1. CLI 提供
2. Web 提供

CLI はローカル検証、回帰確認および補助的利用経路として維持する。
Web は MVP における主要利用導線として位置づける。

この両立により、既存 CLI 資産の価値を維持しつつ、ブラウザからの利用可能性を提供する。

### 3.3 CLI と Web の両立方針

Q-Scout for Spring では、CLI と Web の入口を分離し、内部の解析処理のみを共通化する。

- CLI 入口：`Main` → `CliApplication`
- Web 入口：`WebPageController` / `WebDownloadController` → `WebAnalysisService`
- 共通中核：`SharedAnalysisService`

本方針により、以下を同時に満たす。

- CLI の互換性維持
- Web 固有責務の分離
- 解析コアの再利用
- 将来拡張性の確保

### 3.4 AI 非依存コアの設計方針

Q-Scout for Spring の基本原則は、AI 非依存コアとすることである。

すなわち、解析、スコア算出および人間用レポート生成は、AI なしで成立しなければならない。

AI は価値増幅手段として位置づける。現行 MVP では AI API 自動連携は行わず、AI投入用 Markdown を生成し、ユーザーが外部 AI へ手動投入する方式を採用する。

このため、本システムの中核設計は以下のとおり整理する。

- 解析コアは AI 非依存とする
- 人間用レポートと AI用 Markdown は責務分離する
- AI用 Markdown は補助成果物であり、システム成立の前提とはしない

## 4. システム全体構成

### 4.1 全体アーキテクチャ

Q-Scout for Spring の全体構成は、以下の論理アーキテクチャで表現する。

```text
[Browser]
  ↓
[Web UI]
  ↓
[Web Controller]
  ↓
[WebAnalysisService]
  ↓
[SharedAnalysisService]
  ↓
[ProjectScanner / RuleEngine / ScoreCalculator / ReportGenerator / AiMarkdownGenerator]
  ↓
[Markdown成果物生成]
  ↓
[結果表示 / ダウンロード]
```

CLI 経路は、以下のとおりとする。

```text
[CLI User]
  ↓
[Main]
  ↓
[CliApplication]
  ↓
[SharedAnalysisService]
  ↓
[ProjectScanner / RuleEngine / ScoreCalculator / ReportGenerator / AiMarkdownGenerator]
```

### 4.2 主要サブシステム構成

本システムは、以下のサブシステムに分類して捉える。

1. 入口サブシステム
   - CLI 入口
   - Web 入口

2. Web 実行統括サブシステム
   - アップロード検証
   - zip 展開
   - 一時ディレクトリ管理
   - ダウンロード提供
   - Web 用レスポンス組み立て

3. 共通解析サブシステム
   - プロジェクト走査
   - ルール評価
   - スコア計算
   - レポート生成

4. 成果物管理サブシステム
   - Markdown 出力
   - 一時保存
   - ダウンロード用パス解決

5. 表示・多言語化サブシステム
   - Web UI 文言
   - 人間用レポート文言
   - MessageSource ベースのロケール解決

### 4.3 外部入出力の全体像

外部入力は、以下のとおりである。

- CLI 引数
- Web からの zip アップロード
- 言語切替パラメータ `lang`

外部出力は、以下のとおりである。

- 標準出力サマリ
- Web 画面への結果表示
- `qscout-report.md`
- `qscout-ai-input.md`
- エラーメッセージ
- ダウンロードレスポンス

### 4.4 実行環境前提（Render / Docker / ローカル）

本システムは、以下の前提で動作する。

- Web 実行環境は Render を前提とする
- コンテナはステートレスである
- 永続ストレージは使用しない
- 一時ファイルは `/tmp` を使用する
- Docker により実行環境差異を吸収する
- CLI はローカル環境でも実行可能とする

## 5. 論理構成と責務分割

### 5.1 入口層（CLI / Web）

#### CLI 入口の責務

CLI 入口の責務は、以下のとおりとする。

- `String[] args` の受領
- 引数解析
- `AnalysisRequest` の生成
- 共通解析サービスの呼び出し
- 標準出力サマリ表示
- 終了コード運用

CLI 入口は解析ロジック本体を持たず、CLI 特有の入力と出力を仲介するアダプタ層として扱う。

#### Web 入口の責務

Web 入口の責務は、以下のとおりとする。

- HTTP リクエスト受信
- 画面表示
- multipart アップロード受付
- ダウンロード受付
- Web 向けレスポンスおよび ViewModel の構築
- ユーザー向けエラー表示

Web Controller は薄く保ち、解析本体やファイル展開処理を持たない。

### 5.2 Web オーケストレーション層

WebAnalysisService は、Web 特有の一連処理を統括する。

主な責務は、以下のとおりとする。

- アップロードファイル受領
- 入力検証
- requestId 発行
- 一時ワークスペース準備
- zip 展開
- project root 判定
- `AnalysisRequest` 生成
- SharedAnalysisService 呼び出し
- 結果レスポンス構築
- ダウンロード情報準備
- cleanup 制御

本層は Web 固有の I/O 責務を吸収するが、解析アルゴリズム自体は持たない。

### 5.3 共通解析サービス層

SharedAnalysisService は、CLI と Web の双方から利用される解析中核である。

主な責務は、以下のとおりとする。

- `AnalysisRequest` の受領
- `ProjectScanner` による走査
- `RuleEngine` による分析
- `ScoreCalculator` によるスコア算出
- `ReportGenerator` による人間用レポート生成
- `AiMarkdownGenerator` による AI用 Markdown 生成
- `ReportArtifact` の生成
- 解析結果全体の返却

本層は入口依存の処理を持たず、指定された解析対象に対して成果物を生成する共通サービスとして扱う。

### 5.4 解析コア層

解析コア層は、Q-Scout の価値の中心であり、Web 化後も最大限温存する。

主な構成要素は、以下のとおりである。

- `ProjectScanner`
- `RuleEngine`
- `ScoreCalculator`
- `ReportGenerator`
- `AiMarkdownGenerator`
- `Rule` 群
- ドメイン DTO 群

#### ProjectScanner

- `pom.xml` の存在確認
- `src/main/java` および `src/test/java` の走査
- `ProjectContext` 生成

#### RuleEngine

- 複数ルールの逐次実行
- `AnalysisResult` 構築

#### ScoreCalculator

- 違反重大度に応じた減点計算
- `ScoreSummary` 構築

#### ReportGenerator

- 人間用 Markdown レポート生成

#### AiMarkdownGenerator

- AI投入用 Markdown 生成

#### Rule 群

- 6ルールの候補検出を担う

### 5.5 レポート生成層

レポート生成層は、人間向け成果物と AI向け成果物を分離して扱う。

#### 人間用レポート

目的：
- 開発者および関係者が診断結果を理解するためのレポートとする

特徴：
- ロケールに応じて見出し等を切り替える
- Web および CLI の双方で再利用可能とする
- 人間の読解に最適化する

#### AI用 Markdown

目的：
- 外部 AI へ手動投入するための構造化入力とする

特徴：
- 現行方針では原則英語維持とする
- 人間用レポートとは責務分離する
- AI 非依存コアの補助成果物として扱う

### 5.6 一時ファイル・成果物管理層

本層は、Web 実行時の一時ファイルおよび成果物ファイルを安全に管理する。

主な責務は、以下のとおりである。

- requestId 単位のディレクトリ分離
- upload.zip 保存
- extracted/ 管理
- output/ 管理
- ダウンロード対象の解決
- cleanup

標準ディレクトリ構成は、以下のとおりとする。

```text
/tmp/qscout/{requestId}/
  upload.zip
  extracted/
  output/
```

成果物ファイルは、以下の2種類とする。

- `qscout-report.md`
- `qscout-ai-input.md`

### 5.7 多言語化対応層

多言語化対応層は、主として Web UI と人間用レポートの文言切替を担う。

採用方針は、以下のとおりとする。

- MessageSource ベースとする
- SessionLocaleResolver を採用する
- LocaleChangeInterceptor を採用する
- `?lang=ja` / `?lang=en` により切替可能とする
- デフォルト言語は日本語とする
- AI用 Markdown は原則英語維持とする

本層は解析コアのロジックを変更せず、表示文言側で多言語化を成立させる。

## 6. 主要処理方式

### 6.1 Web 解析の基本フロー

Web の基本フローは、以下のとおりとする。

1. ユーザーが zip をアップロードする
2. Web 入口がリクエストを受け付ける
3. UploadValidationService が入力を検証する
4. TempWorkspaceService が requestId と一時領域を準備する
5. ZipExtractionService が安全に展開する
6. project root を特定する
7. `AnalysisRequest` を生成する
8. SharedAnalysisService を呼び出す
9. 解析、スコア算出および成果物生成を行う
10. WebAnalysisService が画面用レスポンスを組み立てる
11. 結果を画面表示する
12. ダウンロード経路を提供する
13. cleanup 方針に従い後処理を行う

### 6.2 CLI 解析の基本フロー

CLI の基本フローは、以下のとおりとする。

1. `Main` が起動される
2. `CliApplication` が引数を受け取る
3. `ArgumentParser` が `AnalysisRequest` を生成する
4. SharedAnalysisService を呼び出す
5. 解析、スコア算出および成果物生成を行う
6. `ExecutionSummary` を受け取る
7. 標準出力へサマリを表示する
8. エラー時は規定の終了コードで終了する

### 6.3 成果物生成フロー

共通解析サービスは、分析結果に基づき以下の成果物を生成する。

- 人間用レポート
- AI用 Markdown

これらは出力ディレクトリへ保存され、CLI ではパス情報として返却し、Web ではダウンロード対象として扱う。

### 6.4 エラー処理・タイムアウト方針

エラーは、以下の分類で扱う。

- 入力エラー
- 構造エラー
- 実行エラー
- タイムアウト
- 成果物期限切れエラー

方針は、以下のとおりとする。

- 原因をユーザーが理解可能な形で示す
- 解析コア内部の例外をそのまま UI に露出しない
- Web ではユーザー向けメッセージへ変換する
- 60秒超過時は専用の失敗として扱う

### 6.5 クリーンアップ方針

一時ファイルは、ステートレス運用を前提として cleanup する。

基本方針は、以下のとおりとする。

- requestId 単位で分離する
- 処理完了後は不要ファイルを継続保持しない
- エラー時も cleanup を考慮する
- ダウンロード提供との両立のため、成果物保持期間は詳細設計および実装で調整する

## 7. データと成果物の設計方針

### 7.1 主要ドメインモデル

Q-Scout の解析コアでは、以下のドメイン DTO を中核とする。

- `AnalysisRequest`
- `ProjectContext`
- `Violation`
- `Severity`
- `RuleResult`
- `AnalysisResult`
- `ScoreSummary`
- `ReportArtifact`
- `ExecutionSummary`

設計原則は、以下のとおりとする。

- CLI と Web の双方で共通利用する
- 解析コアの意味モデルとして維持する
- 画面表示用 DTO とは分離する

### 7.2 成果物ファイル

本システムの主要成果物は、以下の2つとする。

1. `qscout-report.md`
2. `qscout-ai-input.md`

役割分担は、以下のとおりである。

- `qscout-report.md`
  - 人間向け
  - 要約、違反および改善観点を示す
  - 多言語化対象

- `qscout-ai-input.md`
  - AI投入向け
  - 構造化プロンプト
  - 原則英語維持

### 7.3 画面表示用 DTO / ViewModel の考え方

Web 表示では、ドメイン DTO をそのままビューへ出力せず、必要に応じて Web 用 DTO または ViewModel を用いる。

その理由は、以下のとおりである。

- 表示責務の分離
- エラー表示の整理
- ダウンロードリンク表現の明確化
- 画面要件変更への耐性確保

設計方針は、以下のとおりとする。

- 解析コア DTO は内部表現とする
- 画面向け DTO は表示表現とする
- 両者の責務は明確に分離する

### 7.4 ロケールと言語別出力方針

ロケール適用方針は、以下のとおりとする。

- Web UI は現在選択中のロケールに従う
- 人間用レポートは現在ロケールに従う
- CLI は既定ロケールである日本語を基本とする
- AI用 Markdown は原則英語維持とする

## 8. 採用技術と設計上の制約

### 8.1 採用技術

基本設計上の主要採用技術は、以下のとおりとする。

- Java 17
- Maven
- Spring Boot Web
- Thymeleaf
- MessageSource
- SessionLocaleResolver
- LocaleChangeInterceptor
- Docker
- Render

### 8.2 実行制約

本システムは、以下の制約下で設計する。

- 同期処理
- 最大 60 秒
- zip のみ受付
- 最大 20MB
- `pom.xml` を含む Spring Boot / Maven プロジェクトを前提とする
- ステートレス運用
- 永続ストレージ不使用

### 8.3 セキュリティ前提

入力ファイルは信用しない前提で設計する。

必須方針は、以下のとおりである。

- Zip Slip 防止
- 外部コマンド禁止
- ディレクトリ外アクセス禁止
- 実行ファイルを前提にしない
- Java ソース中心の解析
- 一時ファイルの適切な cleanup

### 8.4 今回スコープ外

以下は、今回の基本設計における前提として未実施または非対象とする。

- JSON 出力
- AI API 自動連携
- 非同期ジョブ
- 進行状況詳細表示
- 認証 / 認可
- 履歴管理
- マルチモジュール対応
- Gradle 対応
- 高度なルール精度改善
- マルチテナント設計

## 9. 現行実装との対応

### 9.1 現行実装で成立済みの事項

現行実装では、少なくとも以下は成立済みである。

- CLI 入口
- Web 入口
- SharedAnalysisService を中心とした共通解析

## 11. 最終結論

Q-Scout for Spring の基本設計は、要件定義で定めた Web MVP の成立条件を満たしつつ、既存 CLI 資産を温存し、共通解析コアを中心に CLI と Web の両入口を両立させる構造として整理する。

本基本設計の中核は、以下のとおりである。

1. 入口を分離する
2. 解析コアを共通化する
3. SharedAnalysisService を解析中核とする
4. Web 固有責務を WebAnalysisService および周辺サービスへ分離する
5. 人間用レポートと AI用 Markdown を分離する
6. ステートレスな `/tmp` 運用を前提とする
7. 多言語化を MessageSource ベースで成立させる
8. AI 非依存コアを維持する

以上により、Q-Scout for Spring は、現行要件、現行実装および詳細設計との境界を保ちながら、`docs/30_基本設計/` に配置すべき正式な骨格設計文書を持つことができる。
