# 提案名：ユーザー向け Figma運用作業手順書 v1.0

## 1. 文書概要

### 文書名
ユーザー向け Figma運用作業手順書

### 版数
v1.0

### 目的
本手順書は、Q-Scout for Spring において、ユーザー自身が Figma 上で Web UI を確認・修正し、その差分を ChatGPT / Codex を通じて安全かつ効率的に実装へ反映するための、ユーザー作業用の標準手順を定義するものである。

本手順書の目的は以下の4点である。

1. UI修正依頼を文章だけに依存しない形へ移行する
2. ユーザーが視覚的に意図を具体化できるようにする
3. ChatGPT / Codex へ渡す修正指示の精度を上げる
4. 実装反映後の確認漏れを防ぐ

---

## 2. 本手順書の位置付け

本手順書は、以下の役割分担を前提とする。

### ユーザー
- Figma 上で UI を確認する
- 必要な見た目修正を Figma 上で具体化する
- 修正意図と優先度を整理する
- ChatGPT へ差分反映依頼を出す
- 反映後の妥当性を最終確認する

### ChatGPT
- Figma 差分を仕様・設計・実装指示へ翻訳する
- 変更影響範囲を整理する
- 必要に応じて Codex 向け実装指示書を作成する
- 多言語化や既存構造との整合を判断する

### Codex
- 実際の HTML / CSS / JS / MessageSource 修正を行う
- 必要ならテスト更新を行う
- 実装結果を報告する

---

## 3. この手順でできること

この手順で主に扱うのは以下である。

1. レイアウト変更
2. セクション順序変更
3. 見出しや説明文の配置変更
4. ボタンやカードの見た目調整
5. 余白・幅・整列の調整
6. 成果物サンプルの見せ方変更
7. ファーストビューの訴求改善
8. 共通 UI 部品の軽微な見た目改善

---

## 4. この手順で直接は扱わないこと

以下は本手順の対象外、または別手順で扱う。

1. 解析ロジック変更
2. ルール定義変更
3. スコア計算変更
4. CLI 機能変更
5. Figma からの自動コード生成
6. 大規模画面追加
7. API 仕様変更
8. バックエンド動作変更が主目的の改修

---

## 5. 作業開始前の前提確認

Figma で修正を始める前に、まず以下を確認する。

### 5.1 対象画面を決める
対象画面を明確にする。

例：
- トップページ
- help ページ
- preview ページ
- rule-help ページ

### 5.2 変更対象セクションを決める
対象セクションを明確にする。

例：
- hero / first view
- 成果物サンプル
- 診断実行エリア
- 結果サマリ
- 言語切替導線
- 補助説明ブロック

### 5.3 変更目的を一文で言える状態にする
必ず、変更目的を短く言語化してから着手する。

例：
- 初見ユーザーに成果物イメージを先に伝えたい
- 診断実行ボタンを視認しやすくしたい
- 画面情報量を整理して読みやすくしたい
- 英語表示でも崩れにくくしたい

## 6. Figmaでの実作業手順

### STEP 1：現状画面を確認する
まず、現在の実装画面を開き、どこを直したいかを確認する。

確認ポイント：
- 情報が多すぎないか
- 何の画面かすぐ分かるか
- 最初に見せたい価値が上に来ているか
- ボタンや導線が目立っているか
- 日本語 / 英語で崩れそうな箇所がないか

### STEP 2：Figma上で対象画面を作る
Figma 上で対象画面を開く、または新規フレームを作る。

推奨：
- 1画面1フレーム
- 変更前を残し、変更後を別フレームにする
- 名前を分かりやすくする

推奨命名例：
- Home_Before
- Home_After
- Help_After_v1
- Preview_After_v1

### STEP 3：変更したい見た目をFigmaで具体化する
修正したい箇所を、文章ではなく Figma 上で見える形にする。

対象例：
- セクション順序の入れ替え
- 見出し位置変更
- カードサイズ変更
- 余白調整
- ボタン位置変更
- 説明文の塊を整理
- 強調ブロック追加

注意：
- 実装可能性を完全に気にしすぎる必要はない
- ただし、まったく別プロダクトになるような大改造は避ける
- まずは「どう見せたいか」を優先してよい

### STEP 4：変更箇所を明示する
Figma 上で、変更箇所が分かるようにしておく。

推奨方法：
- コメントを付ける
- テキストで補足を書く
- セクション名を付ける
- 必要なら番号を振る

推奨記法例：
- 変更1：成果物サンプルをファーストビュー直下へ移動
- 変更2：診断実行エリアを1段下へ移動
- 変更3：ボタン幅を広げ、視認性を改善
- 変更4：説明文を短くし、3点要約に整理

### STEP 5：変更理由を短文で付ける
各変更には、理由を一言添える。

例：
- 初見ユーザーが価値を即理解できるようにするため
- 実行導線が埋もれていたため
- 余白不足で読みづらかったため
- 英語表示時の折り返し余裕を確保するため

## 7. ChatGPTへ依頼する前に整理する情報

ChatGPT へ依頼する前に、最低限以下を整理する。

### 必須項目
1. 対象画面
2. 対象セクション
3. 変更目的
4. 変更箇所一覧
5. 優先度
6. 文言変更の有無
7. 多言語影響の有無

### 推奨項目
8. Before / After の違い
9. できれば維持したい点
10. 今回は触らない点

---

## 8. ChatGPTへ依頼するときの標準手順

### STEP 1：修正対象を伝える
最初に、どの画面の何を変えたいかを明示する。

例：
- トップページの成果物サンプル周辺を修正したい
- help ページの説明ブロックの見せ方を直したい

### STEP 2：Figma差分があることを伝える
ChatGPT に、文章だけでなく Figma を基準にしたいことを伝える。

例：
- Figma で変更案を作ったので、それを基準に実装差分を整理してほしい
- 見た目差分を踏まえた Codex 指示書を作ってほしい

### STEP 3：変更目的を伝える
単なる「こう変えたい」だけでなく、「なぜ変えたいか」を伝える。

例：
- 初見ユーザーへの価値訴求を強めたい
- 見た目の整理と読みやすさを上げたい
- 実行導線の迷いを減らしたい

### STEP 4：優先度を伝える
変更が複数ある場合は優先順位を明示する。

例：
- 最優先：成果物サンプルの位置変更
- 次点：説明文の短文化
- 後回し可：カード装飾の微調整

### STEP 5：ChatGPTに求める成果物を指定する
必要なものを明示する。

例：
- 実装方針メモ
- Codex向け実装指示書
- docs 用仕様書
- 影響範囲整理
- 受け入れ条件

## 9. ChatGPTへの依頼テンプレート

以下をそのまま使ってよい。

【対象画面】
例：トップページ

【対象セクション】
例：成果物サンプル、診断実行エリア、ファーストビュー

【変更目的】
例：初見ユーザーに「何が得られるツールか」をすぐ伝えたい

【変更内容】
1.
2.
3.

【優先度】
例：
- 優先度1：
- 優先度2：
- 優先度3：

【文言変更】
あり / なし

【多言語影響】
あり / なし / 不明

【依頼したい内容】
例：Figma差分を前提に、実装方針整理とCodex向け実装指示書を作成してください

## 10. Codexへ進める判断基準

以下の場合は、ChatGPT で方針整理後、Codex へ進める。

1. HTML / CSS / JS 修正が必要
2. MessageSource 修正が必要
3. テンプレート構造変更が必要
4. 複数ファイルに跨る修正になる
5. テスト更新が必要

逆に、まだ見せ方の方向性が固まっていない場合は、先に ChatGPT で整理する。

---

## 11. 実装後にユーザーが確認すること

Codex 実行後は、必ず以下を確認する。

### 11.1 見た目確認
- Figma の意図に近いか
- 情報の優先順位が改善したか
- 違和感のある余白や配置がないか
- 画面が読みやすくなったか

### 11.2 動作確認
- ボタンが動くか
- 実行導線が壊れていないか
- アップロードが成立するか
- preview / download が壊れていないか
- モーダルや実行中表示が動くか

### 11.3 多言語確認
- 日本語で問題ないか
- 英語でも大きく崩れていないか
- ボタンや見出しが切れていないか

### 11.4 改修範囲確認
- 今回直したかった箇所が改善されたか
- 触らないはずの部分に副作用が出ていないか

## 12. 差し戻し時の進め方

実装結果がイメージと違う場合は、抽象的に「なんか違う」と返さず、以下の形で差し戻す。

### 差し戻し時に書くべきこと
1. どの画面か
2. どのセクションか
3. 何が期待と違うか
4. どう直したいか
5. 優先度は高いか低いか

### 例
- トップページの成果物サンプル領域について、カード間の余白がまだ狭いです
- Figma では3つのカードをもう少し独立して見せたいです
- 見出しは現状のままでよいです
- 余白調整だけ優先度高で再修正してください

## 13. おすすめの最小運用単位

最初から大きくやらず、まずは小さく回す。

推奨する初回対象：
- トップページの1セクションのみ
- 変更点は3つ以内
- 文言変更は最小限
- 動作影響の少ない領域から始める

理由：
- Figma運用の流れを試しやすい
- ChatGPT / Codex との連携が安定しやすい
- 差し戻しが軽く済む
- 手順の改善点を見つけやすい

---

## 14. 初回運用の推奨対象

最初の Figma 実証対象としては、以下を推奨する。

### 第一候補
トップページの成果物サンプル領域

理由：
- 価値訴求に直結する
- 見た目改善の効果が分かりやすい
- 動作影響が比較的小さい
- ポートフォリオ観点でも改善価値が高い

### 第二候補
トップページのファーストビュー

理由：
- 第一印象に効く
- 説明力改善と直結する
- 比較的レイアウト中心で進めやすい

### 第三候補
help ページの説明ブロック

理由：
- 情報整理の改善効果を得やすい
- バックエンド影響が少ない

## 16. 本手順書の最終結論

本手順書の本質は、UI修正を
「ユーザーが頭の中のイメージを文章だけで伝える方式」
から
「ユーザーが Figma で視覚化し、それを ChatGPT / Codex が構造化して反映する方式」
へ移行することにある。

この運用により、
- 修正意図の伝達精度が上がる
- UI改善の試行回数を増やしやすくなる
- テンプレート責務分離済みの実装構造を活かせる
- 将来のポートフォリオ説明にも使える改善履歴を残せる

以上を、Q-Scout for Spring におけるユーザー向け Figma運用の初版標準手順とする。
