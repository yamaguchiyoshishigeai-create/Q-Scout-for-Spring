# AI運用作業スクリプト化候補棚卸し

## 1. 文書目的

本書は、Q-Scout for Spring リポジトリにおいて、現在 AI の判断・操作に依存している運用作業のうち、通常のプログラム的処理へ置き換えられる候補を棚卸しするための整理文書である。

目的は、AI の裁量判断に委ねる必要がない定型作業を `ps1`、`sh`、Python、Java、または GitHub Actions へ段階的に移し、運用ブレ、見落とし、プロンプト作成ミス、Codex トークン消費、ユーザー確認負荷を低減することである。

## 2. 棚卸しの前提

現行リポジトリでは、以下の運用規程が整備されている。

- 改善タスク管理ルール
- ChatGPTリポジトリ編集運用ルール
- Codex連携運用ルール
- CodexPrompt_GitScriptEscalation_Rules
- AIリポジトリ作業証跡管理ルール
- 公開サンプル評価運用方針

また、既存スクリプトとして `scripts/run-sample-evaluation-under-samples.ps1` があり、公開 Spring サンプルの clone、build、Q-Scout 実行、結果集約を一括実行している。

つまり、運用手順の文書化と一部スクリプト化は進んでいるが、改善タスク管理、PR証跡、Codexプロンプト検査、docs整合確認などには、まだ AI 判断・AI 操作に依存している領域が残っている。

## 3. 分類基準

本棚卸しでは、各候補を以下の観点で分類する。

| 観点 | 説明 |
|---|---|
| 現在の実施方法 | AI、ユーザー、Codex、既存スクリプトのどれに依存しているか |
| プログラム化できる範囲 | 機械的に判定・生成・検査できる範囲 |
| AI判断を残す範囲 | 文脈判断、品質判断、優先度判断など、機械化しにくい範囲 |

| 実装難度 | 低 / 中 / 高 |
| Codex要否 | ChatGPT(リポジトリ編集)で可能か、Codex推奨か |
| 後続TSK化 | 独立タスク化すべきか |

## 5. 候補1: TSK台帳整合チェッカー

### 5.1 対象作業

改善タスク一覧、個別 `TSK-***.md`、`解決済み/` フォルダ配置、状態、リンク先、採番の整合確認。

### 5.2 現在の実施方法

ChatGPTが都度、一覧正本と個別ファイルを読み、状態、リンク、配置を判断している。

### 5.3 プログラム化できる範囲

- `改善タスク課題一覧.md` の TSK 行抽出
- 個別 `TSK-***.md` の存在確認
- `解決済み/TSK-***.md` の存在確認
- 一覧の状態と個別ファイルの状態一致確認
- `解決済み` 状態と配置先の一致確認
- 個別ファイルリンクの存在確認
- TSK番号の重複確認
- 最大TSK番号確認
- 登録PRとタスク本義の混同検出補助

### 5.4 AI判断を残す範囲

- 状態不整合が意図的かどうかの最終判断
- 登録PRと実作業PRをどう扱うかの方針判断
- タスク粒度の再設計

### 5.5 推奨実装方式

Python を第一候補とする。Markdown表の解析、ファイル存在確認、レポート出力が容易なため。

### 5.6 優先度

最優先。

### 5.7 後続TSK

`TSK-030 TSK台帳整合チェッカーを実装する` として登録済み。

## 16. 残件登録ゲート

TSK-030、TSK-031、TSK-032 がすべて `解決済み` になった時点で、次の作業へ進む前に本棚卸し文書の残件を再確認する。

再確認対象は以下とする。

- TSK-033 docsリンク・配置規程チェッカー
- TSK-034 Web/CLI/Maven検証支援スクリプト
- TSK-035 i18nキー整合チェッカー
- TSK-036 解決済み化PR生成支援ツール
- TSK-032へ統合候補としたPR本文テンプレート生成支援の扱い

標準処理は以下とする。

1. TSK-030〜TSK-032 の完了結果を確認する。
2. 実装済みチェッカーで新たに自動化しやすくなった領域がないか確認する。
3. 残件候補を `今すぐ登録`、`既存TSKへ統合`、`保留`、`不要` に分類する。
4. `今すぐ登録` と判断したものは、改善タスク管理ルールに従って新規TSKとして登録する。
5. 登録しない候補についても、理由を本棚卸し文書または該当TSKへ追記する。

このゲートを通過するまでは、TSK-029由来のプログラム化計画が完了したものとして扱わない。

## 17. 定期棚卸しゲート

運用手順、横断運用規程、Codex連携規程、改善タスク管理ルール、PR証跡ルール、docs配置ルール、検証手順が追加または修正された場合、次の観点でプログラム化対象を再棚卸しする。

- 新しい手順に、機械的に検査できる条件が含まれているか
- AIが毎回判断している定型作業が増えていないか
- 既存チェッカーへ検査項目として追加できるか
- 新規スクリプトとして分離すべきか
- GitHub Actions化に向くか
- Codexプロンプト作成時に静的検査できるか

再棚卸しのタイミングは以下とする。

- 運用規程を変更するPRを作成したとき
- Codexプロンプト作成ルールを変更したとき
- 改善タスク管理ルールを変更したとき
- PR証跡ルールを変更したとき
- docs配置や命名規約を変更したとき
- 検証手順やサンプル評価手順を変更したとき
- AI判断ミスまたは手順逸脱が発生したとき

再棚卸しの結果、新たなプログラム化候補が見つかった場合は、既存TSKへ統合するか、新規TSKとして登録する。

## 18. 結論

TSK-029 の棚卸し結果として、最初に実装すべき候補は以下の3件である。

1. TSK台帳整合チェッカー
2. CodexプロンプトGit操作準拠チェッカー
3. AI_REPO_RESULT証跡チェッカー

これらは、現行運用で実際に発生した状態誤判定、CodexプロンプトのGit操作制約漏れ、PR証跡の記載漏れリスクを直接減らせる。

また、初期3件で作業が途切れないよう、TSK-030〜TSK-032完了後に残件登録ゲートを実施する。さらに、運用手順が追加または修正された場合は定期棚卸しゲートを通し、プログラム化候補を継続的に発見・登録する。

TSK-029 は、この棚卸し文書、初期実装タスク登録、残件登録ゲート、定期棚卸しゲートが main へ反映され、ユーザー確認が完了した時点で解決済み化を検討できる。
