# 公開サンプル評価運用方針

## 目的

Q-Scout for Spring の診断精度、誤検知傾向、適用可能範囲を継続的に確認するため、公開されている Spring 系サンプルを共通評価対象として扱う。

この文書は、`scripts/run-sample-evaluation-under-samples.ps1` による公開サンプル評価の運用方針と、現時点の採用分類を定義する。

## 基本方針

- 公開サンプル評価は、Q-Scout 本体変更後の比較確認、誤検知確認、構造揺さぶり確認のための補助導線として扱う。
- 1 本のサンプルだけで判断せず、正常系、責務分離系、構造揺さぶり系、軽量技術特性系を組み合わせて確認する。
- build 成功可否と Q-Scout 成功可否は分けて扱う。sample 側事情で build が失敗しても、Q-Scout 評価対象として価値が残る場合は採用候補から直ちに外さない。
- 重量級サンプルは常時完走を必須としない。clone、構成確認、必要最小限の build、Q-Scout 試行までで十分な場合がある。
- 公開サンプルだけでは境界条件の揺さぶりが不足するため、将来的には意図的アンチパターンサンプルの自作も前提とする。

## 実行導線

Windows 環境では、リポジトリルートから次を実行する。

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\run-sample-evaluation-under-samples.ps1
```

主な成果物は以下とする。

- `samples/CodexExec.result`
- `samples/sample-comparison-summary.md`
- `samples/sample-output/<sample-name>/clone.log`
- `samples/sample-output/<sample-name>/build.log`
- `samples/sample-output/<sample-name>/qscout.log`

## 現時点の正式分類

### 中核候補

中核候補は、継続運用時に優先して確認する基準サンプル群とする。

- `spring-petclinic`
  - 正常系ベースライン
- `sample-spring-modulith`
  - 責務分離・モジュール境界確認
- `spring-boot-monolith`
  - 構造揺さぶり確認

### 補助候補

補助候補は、用途別の確認や false-positive check の補強に使う。

- `bookstore`
  - 業務 CRUD / JPA 寄りの参考用
- `gs-rest-service`
  - 軽量 REST 構成確認
- `gs-accessing-data-jpa`
  - JPA / Repository 基本構成確認
- `gs-securing-web`
  - Security 構成確認
- `gs-reactive-rest-service`
  - Reactive / WebFlux 構成確認

### 重量級候補

重量級候補は、常用中核には置かず、必要時だけ使う。

- `spring-petclinic-microservices`
  - 分散構成・高難度確認

### 保留候補

保留候補は、現時点では build または Q-Scout の成立性が不足しており、常用採用を見送る。

- `spring-boot-realworld-example-app`
  - 実務寄り API 構成として有望だが、現時点では build / Q-Scout 成立性に課題あり

## 運用上の読み方

- 中核候補 3 本は、Q-Scout の基準比較対象として優先的に維持する。
- 補助候補は、誤検知確認や特定技術要素の確認用として使い分ける。
- 重量級候補は、通常回帰ではなく節目の確認や性能余裕がある時に使う。
- 保留候補は、環境差異や sample 側前提の見直し後に再評価する。

## 今後の追加方針

今後サンプルを追加する場合は、少なくとも以下のどちらかを満たすことを目安とする。

- より明確な MVC 構成を持つ中規模 Spring Boot アプリ
- マルチモジュールまたは境界分離の比較に有効な構成

加えて、公開サンプルでは十分に得られない境界条件確認のため、将来的に意図的アンチパターンを含む自作サンプルを別系統で整備する。
