# ChatGPTリポジトリ編集運用ルール

## 1. 文書目的

本書は、ChatGPT(リポジトリ編集) がリポジトリ上で小〜中規模の docs 更新、README 更新、局所修正を行う際の作業手順、Pull Request 作成、ユーザー確認、改善タスク状態更新のルールを定義する。

本ルールの目的は、ChatGPT が PR 作成前またはユーザー確認前に改善タスクを過早に `解決済み` と扱うことを防ぎ、ユーザー(人)による最終確認・main merge・外部確認を明確な完了ゲートとして扱うことである。

作業証跡の残し方および PR ブランチ検証先行ルールの詳細については、`AIリポジトリ作業証跡管理ルール.md` を正本とする。

---

## 2. 適用対象

本書は、主に以下の作業へ適用する。

- ChatGPT(リポジトリ編集) による docs 更新
- README 更新
- 小〜中規模ファイルの局所修正
- 改善タスク管理文書の更新
- PR 作成までを ChatGPT が行い、main merge をユーザーが行う作業
- GitHub connector 等によるリポジトリ操作を含む ChatGPT(リポジトリ編集) の証跡記録

以下は原則として Codex 優先とする。

- 実装、複数ファイル更新、テスト実行、build 検証を伴う作業
- 大きい台帳、長大 README、長文設計書、全文取得できないファイルの大規模更新
- 反復修正や実行環境での検証が必要な作業

ただし、Codex 使用量が逼迫している場合、またはユーザーが Codex 温存方針を示した場合は、Codex 優先作業であっても、ChatGPT(会話内)またはChatGPT(リポジトリ編集)で先に分離・前処理できる範囲を確認する。

---

## 3. 作業主体

| 作業主体 | 主な担当 |
|---|---|
| ユーザー(人) | 最終判断、PR確認、外部確認、main への merge、主観的判断、解決済み承認 |
| ChatGPT(会話内) | 方針整理、レビュー、文案作成、作業仕分け |
| ChatGPT(リポジトリ編集) | branch作成、局所編集、PR作成、変更要約、確認観点提示 |
| Codex | 実装、複数ファイル更新、テスト、build、PR準備 |

ChatGPT(リポジトリ編集) は、ユーザー(人)の最終確認を代替してはならない。

---

## 4. 標準作業手順

ChatGPT(リポジトリ編集) で作業する場合は、原則として以下の順序で進める。

1. 対象タスクと関連文書を確認する。
2. 対象ファイルの規模、構造複雑度、全文取得可否を確認する。
3. ChatGPT(リポジトリ編集) で扱える範囲か、Codex へ回すべきかを判定する。
4. Codex へ回す場合でも、ChatGPT(会話内)またはChatGPT(リポジトリ編集)で先に切り出せる方針整理、台帳更新、README / docs確認、小規模設定確認、Codex指示書最小化を検討する。
5. 作業ブランチを作成する。
6. 必要最小限のファイルを編集する。
7. 差分を確認する。
8. PR を作成する。
9. `AIリポジトリ作業証跡管理ルール.md` に従い、PR 本文へ証跡を記録し、必要に応じて PR コメントまたはコミットメッセージ本文へ短縮記録を残す。
10. PR URL、変更ファイル、変更内容、変更対象外、確認観点、テスト実行有無、証跡記録先をユーザーへ提示する。
11. merge 前の動作確認またはローカル確認が必要な場合は、`AIリポジトリ作業証跡管理ルール.md` の PR ブランチ検証先行ルールに従い、対象 PR ブランチ上で確認するよう案内する。
12. ユーザー確認・main merge を待つ。
13. main merge 後、main へ戻して pull した上で最終確認を行うよう案内する。
14. main merge またはユーザーの明示承認を確認した後、必要に応じて改善タスクを `解決済み` へ更新する。

ChatGPT(リポジトリ編集) は、`CodexExec.result` を証跡ファイルとして使用しない。

---

## 5. PRブランチ検証先行ルールとの関係

PR ブランチ検証先行ルールの詳細は、`AIリポジトリ作業証跡管理ルール.md` を正本とする。

ChatGPT(リポジトリ編集) が PR を作成した場合、merge 前の確認は、同正本に従い、原則として対象 PR ブランチ上で行う。main 上での確認は、PR merge 後の最終確認として扱う。

本書では ChatGPT(リポジトリ編集) の作業手順とユーザー案内上の接続のみを定義し、具体的な確認順序・例外条件・証跡記録先は `AIリポジトリ作業証跡管理ルール.md` を参照する。

---

## 6. Codex節約型の事前切り出しゲート

Codex 使用量が少ない場合、またはユーザーが Codex 温存方針を示した場合、ChatGPT(リポジトリ編集)は Codex 依頼前に以下を確認する。

- docs、README、改善タスク台帳、運用規程、小規模設定ファイルの更新をChatGPT(リポジトリ編集)で先に処理できるか。
- 実行検証前に、方針・完了条件・検証対象・除外対象をChatGPT(会話内)で整理できるか。
- Codexへ渡す指示を、最小検証単位または最小修正単位へ分割できるか。
- Codexに渡す必要がある範囲が、build/test、Web起動、CLI起動、ログ切り分け、多数ファイル一括処理、反復修正などに限定されているか。
- Codex実行後に発生した追加課題を改善タスク台帳へ登録する必要があるか。

このゲートにより、Codexを不要に消費する作業を減らし、実行環境を必要とする作業へCodexを集中させる。

## 7. 改善タスク状態更新ルール

### 7.1 原則

ChatGPT(リポジトリ編集) は、PR 作成時点では改善タスクを原則として `解決済み` にしてはならない。

PR 作成時点の状態は、原則として次のいずれかに留める。

- `解決中`
- `確認待ち`

`解決済み/` フォルダへの移動は、ユーザー確認、main merge、またはユーザーの明示的な解決済み承認後に行う。

### 7.2 確認待ちの扱い

`確認待ち` は、対応 PR または変更案は作成済みだが、ユーザー確認、main merge、または外部確認が未完了の状態を示す。

ChatGPT(リポジトリ編集) が PR を作成した時点では、改善タスクは原則として `確認待ち` とする。

### 7.3 解決済み化できる条件

改善タスクを `解決済み` に変更できるのは、原則として以下のいずれかを満たす場合に限る。

- ユーザーが PR または変更内容を確認し、解決済み化を承認した。
- 対象 PR が main へ merge 済みであることを確認した。
- 方針決定のみで完了するタスクについて、ユーザーが方針確定を承認した。
- ユーザーが事前に「PR作成時点で解決済み化まで含めてよい」と明示した。

### 7.4 main merge報告時の後続確認

ユーザーが PR の main merge を報告した場合、ChatGPT(リポジトリ編集) は次タスクへ進む前に、当該 PR に関連する改善タスクの状態を確認する。

確認対象は、以下のいずれかに該当するタスクとする。

- PR本文に `TASK_ID` または `TSK-***` が記載されている。
- 直前の作業で `確認待ち` に更新した。
- PR本文または完了報告で「main merge後に解決済み化が必要」と明記した。
- ユーザーが明示した。

解決済み化が必要な場合は、以下を行う。

1. PR が main merge 済みであることを確認する。
2. 個別 TSK を `解決済み` に更新する。
3. PR番号、main merge済み事実、ユーザー確認結果を記録する。
4. `解決済み/` フォルダへ移動する。
5. `改善タスク課題一覧.md` の状態、根拠、リンク先を更新する。
6. 解決済み化専用 PR を作成した場合は、その PR の merge を待ってから完了扱いとする。

この後続確認が未完了の状態では、原則として次の改善タスクへ進まない。

### 7.5 解決済み化専用PRの例外統合

ChatGPT(リポジトリ編集) は、原則としてユーザー確認および main merge を待つ。

ただし、`docs/00_プロジェクト管理/02_改善タスク管理/改善タスク管理ルール.md` に定める条件をすべて満たす「解決済み化専用PR」に限り、例外的に ChatGPT(リポジトリ編集) が main への統合まで行ってよい。

この例外は、既に main merge 済みまたはユーザー確認済みの元PRを根拠として、改善タスク台帳を整合させる後続処理に限定する。

通常の機能追加、実装修正、テスト追加、CI設定、README本文、設計本文、運用規程本文を変更するPRには適用しない。

例外統合後は、PR状態を再確認し、完了報告へ `merged: true` を記録する。

詳細条件の正本は `docs/00_プロジェクト管理/02_改善タスク管理/改善タスク管理ルール.md` とする。

### 7.6 禁止事項

以下を禁止する。

- ユーザー確認前に改善タスクを `解決済み` にすること。
- ユーザー確認前に `解決済み/` フォルダへ移動すること。
- PR 作成をもって、ユーザー確認または main merge が完了したものとして扱うこと。
- 完了条件が曖昧なまま、状態だけを `解決済み` へ変更すること。
- main merge報告後、関連する `確認待ち` タスクの解決済み化要否を確認せずに次タスクへ進むこと。

## 8. PR本文の必須項目

ChatGPT(リポジトリ編集) が PR を作成する場合、PR本文には原則として以下を含める。

- 目的
- 変更内容
- 変更対象外
- 確認観点
- テスト実行有無
- 必要に応じた PR コメントまたはコミットメッセージ本文の短縮記録有無
- ユーザー確認後に追加の状態更新が必要か
- 改善タスクを `確認待ち` にした場合は、その理由

---

## 9. ユーザーへの完了報告

PR 作成後、ChatGPT はユーザーに次を報告する。

- PR URL
- 作業ブランチ
- 変更ファイル一覧
- 変更内容の要約
- 確認観点
- テスト実行有無
- 証跡記録先
- merge 前確認を PR ブランチ上で行うべきか
- 改善タスクの現在状態
- ユーザー確認または main merge 後に必要な後続処理

この報告では、ユーザー確認前の作業を `解決済み` と断定してはならない。

---

## 10. 例外運用

以下の場合は、ユーザー確認前に `解決済み` へ進められることがある。

- ユーザーが事前に、対象タスクの解決済み化まで ChatGPT(リポジトリ編集) に明示委任した場合。
- 文書上の方針確定だけで完了するタスクについて、会話内でユーザーが方針確定を承認した場合。
- PRなしで main へ直接反映することをユーザーが明示指示した軽微修正の場合。

ただし、例外を適用する場合も、PR本文または完了報告にその理由を明記する。

---

## 11. 改善タスク管理ルールとの関係

本ルールは、ChatGPT(リポジトリ編集) における作業完了ゲートを定義する。

改善タスクの状態定義、採番、解決済み移動手順は `docs/00_プロジェクト管理/02_改善タスク管理/改善タスク管理ルール.md` を正本とする。

ただし、ChatGPT(リポジトリ編集) では、解決済み移動の前に本書で定めるユーザー確認ゲートを必須とする。
