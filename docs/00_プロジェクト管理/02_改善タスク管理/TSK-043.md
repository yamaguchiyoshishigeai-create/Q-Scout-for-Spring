# TSK-043 改善タスク台帳の正本・生成物分離を設計する

- 区分: 運用 / docs / 自動化 / 課題管理
- 件名: 改善タスク台帳の正本・生成物分離を設計する
- 状態: 確認待ち
- 優先度: 高
- 実施主体: ChatGPT(会話内) → ChatGPT(リポジトリ編集) / Codex
- 関連箇所: `改善タスク課題一覧.md`, 個別 `TSK-***.md`, `解決済み/TSK-***.md`, `scripts/check_tsk_registry.py`, 将来の `scripts/generate_tsk_registry.py`

## 発端

`改善タスク課題一覧.md` は、TSK-043まで登録された時点で、実質的にSQLテーブルに近い構造を持つ巨大なMarkdown表になっている。

現行の一覧は、人間が俯瞰しやすい一方で、ChatGPT(リポジトリ編集)が更新する場合はファイル全文に近い置換操作になりやすく、局所修正のつもりでも無関係行へ混入変更が発生するリスクが高まりつつある。

直近のTSK追加・解決済み化作業でも、一覧の一部既存行に軽微な文言混入が発生しかけ、追加修正で戻した。

このため、単純な一覧分割だけでなく、個別 `TSK-***.md` を正本レコード、`改善タスク課題一覧.md` を生成物または索引用ファイルとして扱う方式を検討する必要がある。

## 目的

改善タスク台帳の更新安全性と一覧性を両立するため、台帳の正本と生成物の役割分担を設計する。

特に、ChatGPT(リポジトリ編集)が巨大Markdown表を全文更新し続ける運用を避け、1タスク1ファイル単位の局所更新を基本にする。

## 現行構造の整理

### 1. `改善タスク課題一覧.md` が保持している情報

現行の一覧は、以下の列を持つ。

| 列 | 役割 | 正本性の評価 |
|---|---|---|
| 管理ID | TSK番号 | 個別TSKと重複するが索引上は必要 |
| 区分 | タスク分類 | 個別TSKの先頭メタデータに寄せられる |
| 件名 | タスク名 | 個別TSKのタイトル・件名に寄せられる |
| 状態 | 未解決 / 解決中 / 確認待ち / 解決済み / 保留 | 個別TSKの先頭メタデータに寄せられる |
| 優先度 | 最優先 / 高 / 中 / 低 | 個別TSKの先頭メタデータに寄せられる |
| 実施主体 | 作業分類 | 個別TSKの先頭メタデータに寄せられる |
| 関連箇所 | 関連ファイル・領域 | 個別TSKの先頭メタデータまたは本文に寄せられる |
| 概要 | 一覧用短文 | 個別TSK本文から生成または手動要約として保持可能 |
| 完了条件 | 完了判定 | 個別TSK本文の完了条件に寄せられる |
| 根拠 / 関連PR / 備考 | 履歴・根拠 | 個別TSK本文の関連PR・備考に寄せられる |
| 個別ファイル | リンク | 生成可能 |

一覧の多くの列は個別TSKと重複しており、将来的には一覧を「正本」ではなく「派生ビュー」として扱える余地が大きい。

### 2. 個別 `TSK-***.md` が保持している情報

個別TSKは、先頭メタデータと本文を持つ。

先頭メタデータは概ね以下である。

- 区分
- 件名
- 状態
- 優先度
- 実施主体
- 関連箇所

本文は概ね以下である。

- 発端
- 目的または現状認識
- 対応方針
- 完了条件
- 関連ファイル
- 関連PR
- 備考

この構造は、1ファイル1レコードとして扱いやすい。

## 正本候補の比較

### 案A: 個別TSK Markdown正本 + 一覧生成物/索引化

個別 `TSK-***.md` および `解決済み/TSK-***.md` を正本とし、`改善タスク課題一覧.md` は生成物または索引用ファイルに寄せる。

#### 利点

- 既存資産を最大限活かせる。
- ChatGPT(リポジトリ編集)が1タスク1ファイルを局所更新できる。
- GitHub上の差分が読みやすい。
- Markdownのまま人間が読める。
- 巨大表の全文更新頻度を下げられる。

#### 欠点

- 先頭メタデータ記法の揺れを統一する必要がある。
- 一覧生成スクリプトが必要になる。
- 現行 `scripts/check_tsk_registry.py` の前提を段階的に変える必要がある。

#### 評価

最有力案とする。

### 案B: YAML / JSON正本 + Markdown生成

各TSKをYAMLまたはJSONで保持し、Markdown本文や一覧を生成する。

#### 利点

- 機械処理しやすい。
- CRUD構造が明確になる。
- 一覧・検索・集計に強い。

#### 欠点

- 既存Markdown資産と二重管理になりやすい。
- 人間が読む文書としてはMarkdownより弱い。
- 移行コストが高い。
- ChatGPT(リポジトリ編集)ではJSON/YAMLの構文破損リスクもある。

#### 評価

将来候補に留める。現時点では採用しない。

### 案C: 現行Markdown一覧正本 + 番号レンジ別分割

`改善タスク課題一覧_TSK-001-025.md`、`改善タスク課題一覧_TSK-026-050.md` のように番号レンジで分割する。

#### 利点

- 現行構造を大きく変えずにファイルサイズを下げられる。
- 状態変更時にファイル間移動が不要で、状態別分割より安全。
- ChatGPT(リポジトリ編集)の編集対象を小さくできる。

#### 欠点

- 正本が依然としてMarkdown表であり、手編集リスクは残る。
- チェッカーを分割台帳対応にする必要がある。
- 一覧性を補う索引ファイルが必要になる。

#### 評価

案Aへの移行前の暫定策、または案Aと併用する生成物の分割方式として有効。

### 案D: 現行Markdown一覧正本を継続

現行どおり `改善タスク課題一覧.md` を単一正本として維持する。

#### 利点

- 変更量が最小。
- 現行チェッカーをそのまま使える。

#### 欠点

- TSK増加に比例して全文更新リスクが高まる。
- ChatGPT(リポジトリ編集)が無関係行を壊すリスクが増える。
- 将来的にファイル取得・更新の失敗や差分肥大が起きやすい。

#### 評価

短期維持は可能だが、長期方針としては非推奨。

## 採用方針

本タスクでは、以下を採用方針とする。

**個別 `TSK-***.md` / `解決済み/TSK-***.md` を正本候補とし、`改善タスク課題一覧.md` は段階的に生成物または索引へ移行する。**

ただし、即時に一覧正本を廃止しない。

移行期間中は、以下の扱いとする。

- 現行の `改善タスク課題一覧.md` は引き続き運用上の一覧正本として扱う。
- 新規TSK追加・状態変更・解決済み化では、従来どおり一覧と個別TSKの整合を取る。
- ただし、将来の正本候補は個別TSKファイル群と定める。
- 一覧の生成物化、分割、索引化は後続TSKで段階的に実施する。

## 生成スクリプト方針

将来的に `scripts/generate_tsk_registry.py` を追加する。

### 入力

- `docs/00_プロジェクト管理/02_改善タスク管理/TSK-*.md`
- `docs/00_プロジェクト管理/02_改善タスク管理/解決済み/TSK-*.md`

### 抽出対象

- 管理ID
- 区分
- 件名
- 状態
- 優先度
- 実施主体
- 関連箇所
- 概要
- 完了条件
- 根拠 / 関連PR / 備考
- 個別ファイルリンク

### 出力候補

- `改善タスク課題一覧.md`
- 番号レンジ別の分割一覧
- 状態別サマリ
- チェック用の一時生成結果

### 実行モード候補

- `generate`: 一覧ファイルを生成・上書きする
- `check`: 生成結果と既存一覧の差分を検査する
- `summary`: 状態別件数・最大TSK番号を出力する

## 既存チェッカーとの責務分担

### 現行 `check_tsk_registry.py`

当面は以下を担当する。

- 一覧と個別TSKの採番整合
- 一覧と個別TSKの状態整合
- 通常配置 / 解決済み配置の整合
- リンク先の存在確認
- 最大TSK番号・行数の確認

### 将来の `generate_tsk_registry.py`

将来的には以下を担当する。

- 個別TSKから一覧行を生成する
- 一覧の並び順を正規化する
- 状態別・番号別の派生ビューを生成する
- 生成結果と既存一覧の差分を検査する

### 将来の役割変更

正本・生成物分離後は、`check_tsk_registry.py` を以下のように再定義する。

- 個別TSKメタデータの必須項目チェック
- 個別TSKの配置と状態の整合チェック
- 生成結果とコミット済み一覧の一致チェック
- 一覧を直接正本としてではなく、生成物として検査する

## 段階移行手順

### Phase 1: 方針確定

- 本タスクで正本候補と生成物化方針を確定する。
- 既存運用は壊さない。
- 一覧正本の即時廃止は行わない。

### Phase 2: 個別TSKメタデータ記法統一

- 個別TSK先頭メタデータの必須項目を定義する。
- 既存TSKの記法揺れを棚卸しする。
- 生成に必要な項目が不足していれば補正方針を決める。

### Phase 3: 生成スクリプト仕様化

- `generate_tsk_registry.py` の入力、出力、実行モード、失敗条件を定義する。
- 最初は `check` モードのみでもよい。
- 生成結果をコミットするか、一時生成して差分検査するかを決める。

### Phase 4: チェッカー拡張

- `check_tsk_registry.py` を生成物検証に対応させる。
- 必要に応じて `test_generate_tsk_registry.py` を追加する。
- 既存の9/9回帰テスト相当を維持する。

### Phase 5: 一覧生成物化または索引化

- 生成結果が安定してから、`改善タスク課題一覧.md` を生成物または索引へ寄せる。
- 必要に応じて番号レンジ別分割を採用する。
- ChatGPT(リポジトリ編集)による直接の巨大表更新を原則禁止または例外扱いにする。

## 備考

本タスクは、単なる台帳分割ではなく、改善タスク管理を「Markdown表の手編集」から「個別TSK正本 + 一覧生成または索引」へ移行できるかを検討するための設計タスクである。

実際の生成スクリプト実装、チェッカー改修、既存一覧の分割・生成物化は、本タスクで方針を固めた後に後続TSKとして扱う。
