# 改善タスク管理ルール

## 1. 文書目的
本ルールは、Q-Scout for Spring における改善タスクを、小粒度で安全に更新できる文書構成で管理するための正式ルールを定義する。

改善タスク管理では、一覧性と履歴性を両立しつつ、ChatGPT(リポジトリ編集) と Codex の双方で局所更新しやすいことを重視する。

本管理は、主として以下を対象とする。

- 外部評価を踏まえた改善
- README / UI / UX / 訴求文の改善
- docs 運用改善
- 実装方針に基づく継続改善
- 品質向上のための整理済み改善タスク

## 2. 文書構成と役割分担

### 2.1 固定ルール文書
`改善タスク管理ルール.md` は、改善タスク運用の固定ルール正本とする。

### 2.2 一覧文書
`改善タスク課題一覧.md` は、改善タスク全体を俯瞰する一覧正本とする。

- 全 `TSK-***` の状態、優先度、実施主体、関連箇所、概要、完了条件、根拠 / 備考を記載する
- 解決済みタスクも一覧から削除しない
- 個別ファイルリンクを必ず保持する

### 2.3 個別タスク文書
`TSK-***.md` は、各改善タスクの詳細正本とする。

- 件名、状態、優先度、実施主体、関連箇所、発端、現状認識、対応方針、完了条件、関連ファイル、関連PR、備考を記載する
- 個別更新は原則として対象 `TSK-***.md` と必要最小限の一覧更新に留める

## 3. 運用ルール
- 改善項目は、原則として `改善タスク課題一覧.md` と対応する `TSK-***.md` で管理する
- 各項目には一意の管理IDを付与する
- 状態は少なくとも `未解決`、`解決中`、`確認待ち`、`解決済み`、`保留` のいずれかで管理する
- 解決済みにした項目は、必要に応じて根拠ファイルや PR 番号を記録する
- 大規模な実装作業に分解する場合は、関連文書や Pull Request へリンクを張る
- 履歴を残すため、項目削除ではなく状態更新を原則とする
- 既存内容を落とさず、Markdown リンクで追跡できる状態を維持する
- ChatGPT(リポジトリ編集) が PR を作成した時点では、原則として `解決済み` ではなく `確認待ち` として扱う

## 4. 会話中課題の登録ゲート

会話中に新しい改善課題、運用課題、検証課題、将来対応案が発生した場合、課題が流れて消えることを防ぐため、以下の登録ゲートを適用する。

### 4.1 登録対象とする課題

以下のいずれかに該当するものは、原則として改善タスク登録対象とする。

- 後続作業として実施する可能性が高い改善案
- docs、README、UI、テスト、依存更新、運用規程、公開サンプル評価に影響する課題
- ChatGPT(リポジトリ編集) または Codex に作業依頼する可能性がある課題
- ユーザーが「後で対応したい」「課題として残したい」「運用規律化したい」と示した課題
- 既存タスクの完了条件から外れた派生課題
- 作業中に発見された既存不具合、未検証事項、リンク切れ、台帳不整合、PR未処理事項

### 4.2 登録対象外としてよいもの

以下は、原則として改善タスク登録を省略してよい。

- その場で回答のみで完結した説明
- 実施予定のない雑談的な案
- 既存TSKに明確に包含され、個別管理する必要がない補足
- 一時的な確認で、後続作業や再確認が不要なもの

ただし、後で再参照する可能性がある場合は、既存TSKの備考または関連PR本文に記録する。

### 4.3 棚卸しタイミング

以下のタイミングでは、未登録課題の有無を確認する。

- まとまった方針決定を行った後
- ChatGPT(リポジトリ編集) がPRを作成した後
- Codex実行結果を確認した後
- 既存PRのmerge / close / recreate判断を行った後
- 新しい運用ルールや分離方針を決めた後
- 会話が長くなり、論点が複数に分岐したと判断したとき

### 4.4 登録手順

新規登録が必要な場合は、以下の順で処理する。

1. `改善タスク課題一覧.md` で最大 `TSK-***` を確認する。
2. 最大番号 + 1 で新規管理IDを採番する。
3. `TSK-***.md` を作成する。
4. 発端、現状認識、対応方針、完了条件、関連ファイル、関連PRを記載する。
5. `改善タスク課題一覧.md` へ一覧行を追加する。
6. 作業PRを作成する場合は、PR本文へ追加タスクの概要を記録する。

### 4.5 既存TSKへの統合判断

新規課題が既存TSKに強く含まれる場合は、新規採番せず、既存TSKへ追記してよい。

ただし、以下に該当する場合は新規TSKとして分離する。

- 完了条件が既存TSKと異なる
- 実施主体が異なる
- 実施時期または優先度が大きく異なる
- 既存TSKに統合すると粒度が大きくなりすぎる
- 将来、独立してPR化・Codex依頼・検証する可能性が高い

### 4.6 main merge後の解決済み化ゲート

ユーザーが「mainへmergeした」「問題なしとしてmergeした」「PRをmergeした」と報告した場合、ChatGPT は次の作業へ進む前に、関連する改善タスクの解決済み化要否を確認する。

このゲートは、`確認待ち` のまま残るタスクを防止するための標準処理である。

確認対象は以下とする。

- merge された PR 本文に `TASK_ID`、`TSK-***`、改善タスク番号が記載されているタスク
- 直前の会話で `確認待ち` に更新したタスク
- PR 本文または PR コメントで「main merge 後に解決済み化」と記録したタスク
- ユーザーが明示したタスク

標準処理は以下とする。

1. 対象 PR が main へ merge 済みであることを確認する。
2. 関連 `TSK-***.md` の状態を確認する。
3. 解決済み化条件を満たす場合、個別ファイルを `解決済み` に更新する。
4. 根拠として PR 番号、main merge 済み事実、ユーザー確認結果を追記する。
5. 個別ファイルを `解決済み/` フォルダへ移動する。
6. `改善タスク課題一覧.md` の状態、根拠、リンク先を更新する。
7. 解決済み化専用 PR を作成した場合、その PR 自体は `確認待ち` として扱い、ユーザー確認または main merge 後に完了扱いとする。

ただし、ユーザーが「次タスクへ進む前に確認待ちの処理を行うべき」と明示した場合は、必ずこのゲートを先に処理する。

## 5. ステータス定義

### 5.1 未解決
課題認識済みだが、未着手の状態。

### 5.2 解決中
対応方針が定まり、作業着手済みの状態。

### 5.3 確認待ち
対応 PR または変更案は作成済みだが、ユーザー確認、main merge、または外部確認が未完了の状態。

ChatGPT(リポジトリ編集) または Codex が PR を作成した時点では、原則としてこの状態に留める。

### 5.4 解決済み
リポジトリ反映、確認、または方針確定まで完了した状態。

ただし、ChatGPT(リポジトリ編集) が関与するタスクでは、原則としてユーザー確認、main merge、またはユーザーの明示的な解決済み承認を経た後に限り `解決済み` とする。

### 5.5 保留
課題認識はあるが、優先度・前提条件・依存関係等により当面着手しない状態。

## 6. 優先度基準

### 6.1 最優先
対外評価、主要導線、利用継続性、重大な誤解、または実利用に強く影響する改善。

### 6.2 高
見せ方、説明力、保守性、主要画面品質に大きく効く改善。

### 6.3 中
価値はあるが、直近の主要評価や主要導線には直結しない改善。

### 6.4 低
将来拡張や長期的整理として有効だが、直近優先度は低い改善。

## 7. 区分基準
改善項目の区分は、必要に応じて以下を用いる。

- README
- UI / UX
- 多言語化
- docs
- 実装
- テスト
- セキュリティ
- 公開サンプル評価
- ポートフォリオ訴求
- 運用

## 8. 実施主体の記載基準
実施主体は、現行の作業先分類に従って記載する。

- ユーザー(人)
- ChatGPT(会話内)
- ChatGPT(リポジトリ編集)
- Codex

必要に応じて、主担当 / 補助担当を併記してよい。

## 9. 採番ルール
- 管理ID は `TSK-001` 形式とする
- 新規追加は最大番号 + 1 を採用する
- 重複採番は禁止とする
- 欠番は原則として再利用しない
- 状態変更や `解決済み/` への移動では管理IDを変更しない

## 10. 解決済み移動ルール

### 10.1 初回配置ルール
- 状態が `未解決`、`解決中`、`確認待ち`、`保留` のタスクは通常フォルダに配置する
- 状態が `解決済み` のタスクは `解決済み/` に配置する

### 10.2 解決済み化の前提条件

解決済み化の前提として、原則として以下のいずれかを満たすこと。

- ユーザーが PR または変更内容を確認し、解決済み化を承認した
- 対象 PR が main へ merge 済みであることを確認した
- 方針決定のみで完了するタスクについて、ユーザーが方針確定を承認した
- ユーザーが事前に、PR 作成時点で解決済み化まで含めてよいと明示した

ChatGPT(リポジトリ編集) が PR を作成しただけの段階では、原則として解決済み化してはならない。

### 10.3 解決済み化の順序
1. 解決済み化の前提条件を満たしていることを確認する
2. 個別ファイルの状態を `解決済み` に更新する
3. 根拠となる PR / commit / 備考を追記する
4. `解決済み/` フォルダへ移動する
5. 一覧文書のリンク先を `解決済み/TSK-***.md` に更新する

### 10.4 一覧での扱い
- 解決済みタスクは一覧から削除しない
- 一覧では状態を `解決済み` として保持する
- 履歴検索可能な状態を維持する

### 10.5 解決済み化専用PRの扱い

main merge 後の後続処理として、`確認待ち` タスクを `解決済み` へ移動するためだけの PR を作成した場合、その PR は解決済み化の実務処理である。

ただし、その PR 自体が未 merge の間は、当該処理はまだ main に反映されていないため、完了報告では次のように扱う。

- PR 作成後: 「解決済み化 PR 作成済み / ユーザー確認待ち」
- PR main merge 後: 「解決済み化完了」

この区別により、解決済み化作業そのものの確認待ち状態を明確にする。

### 10.6 解決済み化専用PRのChatGPT自律merge条件

ChatGPT(リポジトリ編集) は、以下の条件をすべて満たす場合に限り、解決済み化専用PRを自律的に main へ merge してよい。

1. ユーザーが対象タスクについて「解決済み化してください」または同等の指示を明示している。
2. 解決済み化の根拠となる元PRが `merged: true` であることを確認済みである。
3. 解決済み化専用PRの変更対象が、以下の範囲に限定されている。
   - `解決済み/TSK-***.md` の追加
   - 通常配置側 `TSK-***.md` の削除
   - `改善タスク課題一覧.md` の状態、根拠、リンク先更新
4. 実装コード、設定ファイル、README本文、設計本文、運用規程本文を変更していない。
5. PR本文に、ChatGPTが自律merge可能な解決済み化専用PRであること、根拠PR、確認結果、変更対象外を明記している。
6. merge後にPR状態を再確認し、完了報告で `merged: true` を確認したことを明示する。

この条件を満たす解決済み化専用PRでは、ユーザーへ追加の merge 操作を依頼せず、ChatGPT(リポジトリ編集) が merge まで実行してよい。

ただし、次のいずれかに該当する場合は自律mergeしてはならない。

- 解決済み化以外の文書本文変更を含む。
- 実装コード、設定ファイル、CI、README本文、設計文書、運用規程本文を変更している。
- PRに競合、merge不可、CI失敗、レビュー未解決、または権限エラーがある。
- ユーザーが明示的に merge を保留している。
- 変更内容に判断余地がある。

## 11. 再オープン時の戻しルール
- `解決済み/TSK-***.md` を通常フォルダへ戻す
- 一覧文書の状態を `未解決`、`解決中`、`確認待ち`、`保留` のいずれかへ戻す
- 一覧文書のリンク先も通常フォルダ側へ戻す
- 再オープン時も管理IDは変更しない

## 12. 更新単位の原則
- 状態変更のみの場合も、一覧文書と個別タスク文書の整合を取る
- ルール変更時のみ `改善タスク管理ルール.md` を更新する
- 一覧性を落とさない範囲で、個別詳細は各 `TSK-***.md` に寄せる
