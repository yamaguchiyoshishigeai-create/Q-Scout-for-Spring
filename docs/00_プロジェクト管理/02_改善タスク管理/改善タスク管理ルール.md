# 改善タスク管理ルール

## 1. 文書目的
本ルールは、Q-Scout for Spring における改善タスクを、小粒度で安全に更新できる文書構成で管理するための正式ルールを定義する。

改善タスク管理では、一覧性と履歴性を両立しつつ、ChatGPT(リポジトリ編集) と Codex の双方で局所更新しやすいことを重視する。

本管理は、主として以下を対象とする。

- 外部評価を踏まえた改善
- README / UI / UX / 訴求文の改善
- docs 運用改善
- 実装方針に基づく継続改善
- 品質向上のための整理済み改善タスク

## 2. 文書構成と役割分担

### 2.1 固定ルール文書
`改善タスク管理ルール.md` は、改善タスク運用の固定ルール正本とする。

### 2.2 一覧文書
`改善タスク課題一覧.md` は、改善タスク全体を俯瞰する一覧正本とする。

- 全 `TSK-***` の状態、優先度、実施主体、関連箇所、概要、完了条件、根拠 / 備考を記載する
- 解決済みタスクも一覧から削除しない
- 個別ファイルリンクを必ず保持する

### 2.3 個別タスク文書
`TSK-***.md` は、各改善タスクの詳細正本とする。

- 件名、状態、優先度、実施主体、関連箇所、発端、現状認識、対応方針、完了条件、関連ファイル、関連PR、備考を記載する
- 個別更新は原則として対象 `TSK-***.md` と必要最小限の一覧更新に留める

## 3. 運用ルール
- 改善項目は、原則として `改善タスク課題一覧.md` と対応する `TSK-***.md` で管理する
- 各項目には一意の管理IDを付与する
- 状態は少なくとも `未解決`、`解決中`、`解決済み`、`保留` のいずれかで管理する
- 解決済みにした項目は、必要に応じて根拠ファイルや PR 番号を記録する
- 大規模な実装作業に分解する場合は、関連文書や Pull Request へリンクを張る
- 履歴を残すため、項目削除ではなく状態更新を原則とする
- 既存内容を落とさず、Markdown リンクで追跡できる状態を維持する

## 4. ステータス定義

### 4.1 未解決
課題認識済みだが、未着手の状態。

### 4.2 解決中
対応方針が定まり、作業着手済みの状態。

### 4.3 解決済み
リポジトリ反映、確認、または方針確定まで完了した状態。

### 4.4 保留
課題認識はあるが、優先度・前提条件・依存関係等により当面着手しない状態。

## 5. 優先度基準

### 5.1 最優先
対外評価、主要導線、利用継続性、重大な誤解、または実利用に強く影響する改善。

### 5.2 高
見せ方、説明力、保守性、主要画面品質に大きく効く改善。

### 5.3 中
価値はあるが、直近の主要評価や主要導線には直結しない改善。

### 5.4 低
将来拡張や長期的整理として有効だが、直近優先度は低い改善。

## 6. 区分基準
改善項目の区分は、必要に応じて以下を用いる。

- README
- UI / UX
- 多言語化
- docs
- 実装
- テスト
- セキュリティ
- 公開サンプル評価
- ポートフォリオ訴求
- 運用

## 7. 実施主体の記載基準
実施主体は、現行の作業先分類に従って記載する。

- ユーザー(人)
- ChatGPT(会話内)
- ChatGPT(リポジトリ編集)
- Codex

必要に応じて、主担当 / 補助担当を併記してよい。

## 8. 採番ルール
- 管理ID は `TSK-001` 形式とする
- 新規追加は最大番号 + 1 を採用する
- 重複採番は禁止とする
- 欠番は原則として再利用しない
- 状態変更や `解決済み/` への移動では管理IDを変更しない

## 9. 解決済み移動ルール

### 9.1 初回配置ルール
- 状態が `未解決`、`解決中`、`保留` のタスクは通常フォルダに配置する
- 状態が `解決済み` のタスクは `解決済み/` に配置する

### 9.2 解決済み化の順序
1. 個別ファイルの状態を `解決済み` に更新する
2. 根拠となる PR / commit / 備考を追記する
3. `解決済み/` フォルダへ移動する
4. 一覧文書のリンク先を `解決済み/TSK-***.md` に更新する

### 9.3 一覧での扱い
- 解決済みタスクは一覧から削除しない
- 一覧では状態を `解決済み` として保持する
- 履歴検索可能な状態を維持する

## 10. 再オープン時の戻しルール
- `解決済み/TSK-***.md` を通常フォルダへ戻す
- 一覧文書の状態を `未解決`、`解決中`、`保留` のいずれかへ戻す
- 一覧文書のリンク先も通常フォルダ側へ戻す
- 再オープン時も管理IDは変更しない

## 11. 更新単位の原則
- 状態変更のみの場合も、一覧文書と個別タスク文書の整合を取る
- ルール変更時のみ `改善タスク管理ルール.md` を更新する
- 一覧性を落とさない範囲で、個別詳細は各 `TSK-***.md` に寄せる
