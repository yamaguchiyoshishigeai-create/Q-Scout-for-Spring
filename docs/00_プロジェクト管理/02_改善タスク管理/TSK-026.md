# TSK-026 確認待ちタスクの解決済み化・移動漏れを防止する

- 区分: 運用 / docs
- 件名: 確認待ちタスクの解決済み化・移動漏れを防止する
- 状態: 確認待ち
- 優先度: 高
- 実施主体: ChatGPT(会話内) → ChatGPT(リポジトリ編集)
- 関連箇所: 改善タスク管理ルール, 改善タスク課題一覧, `解決済み/` フォルダ, ChatGPTリポジトリ編集運用ルール

## 発端
ChatGPT(リポジトリ編集) が改善タスク対応PRを作成した時点で、該当タスクを `確認待ち` に更新する運用は整備された。一方で、ユーザーがPR内容を確認し main へ merge した後、該当 `TSK-***.md` を `解決済み` に更新して `解決済み/` フォルダへ移動する作業が自動化されていない。

そのため、実際には解決済みであるにもかかわらず、台帳上は `確認待ち` のまま残るタスクが発生している可能性がある。

## 現状認識
現行ルールでは、ChatGPT(リポジトリ編集) がPRを作成した時点では `確認待ち` に留め、ユーザー確認または main merge 後に `解決済み` 化することが定義されている。

しかし、main merge 後に以下を行う標準手順・棚卸し手順が弱い。

- 個別 `TSK-***.md` の状態を `解決済み` へ更新する
- 根拠PR番号・merge済み事実を追記する
- `解決済み/` フォルダへ移動する
- `改善タスク課題一覧.md` の状態・リンク先を更新する
- `確認待ち` のまま残っているタスクを定期的に棚卸しする

PR #50 / PR #51 と TSK-018 の流れで、確認待ち後にユーザー確認・main merge・解決済み化PRを挟む運用が実例化されたため、この流れを標準ゲートとして文書化する。

## 対応方針
以下を検討・整備する。

1. `確認待ち` タスクの棚卸しを実施し、main merge済みで解決済み化できるタスクを抽出する。
2. 解決済み化可能なタスクについて、個別ファイルの状態を `解決済み` に更新する。
3. 根拠PR、merge済み確認、関連コミットを個別ファイルと一覧へ記録する。
4. 対象ファイルを `解決済み/` フォルダへ移動する。
5. `改善タスク課題一覧.md` の状態とリンク先を更新する。
6. 今後、ユーザーが「mainへmergeした」と報告した時点で、関連TSKの解決済み化候補を確認する運用ゲートを明文化する。

## 実施内容

- `改善タスク管理ルール.md` に main merge後の解決済み化ゲートを追加。
- `改善タスク管理ルール.md` に解決済み化専用PRの扱いを追加。
- `ChatGPTリポジトリ編集運用ルール.md` に main merge報告時の後続確認を追加。
- `ChatGPTリポジトリ編集運用ルール.md` に、確認待ちタスクの解決済み化要否を確認せず次タスクへ進むことの禁止事項を追加。
- 既存台帳では、PR #51 main merge により TSK-018 は解決済み化済みであることを確認。
- 追加で解決済み化可能な `確認待ち` タスクは、現時点では検出されていない。

## 解決策案

### 案A: 手動棚卸しPR方式

ChatGPT(リポジトリ編集) が定期的またはユーザー報告後に `確認待ち` タスクを棚卸しし、解決済み化可能なものをまとめてPR化する。

- 利点: GitHub Actions等の追加実装が不要
- 欠点: ユーザー報告または定期確認が必要

### 案B: PR merge後チェックリスト方式

PR作成時またはmerge後報告時に、関連TSKの状態更新が必要かを必ず確認するチェックリストを運用規程へ追加する。

- 利点: 既存運用に組み込みやすい
- 欠点: 完全自動ではない

### 案C: GitHub Actions等による半自動検出

PR本文の `TASK_ID` や関連TSK番号をもとに、merge済みPRと `確認待ち` タスクの不整合を検出する仕組みを検討する。

- 利点: 漏れ検出を自動化できる可能性がある
- 欠点: 実装・保守・誤検出対応が必要であり、現時点ではCodexまたは追加検証が必要

現時点では、Codex使用量を温存する方針も踏まえ、案Aと案Bを先行し、必要になれば案Cを将来検討とする。

## 完了条件
- `確認待ち` のまま残っているタスクが棚卸しされる。
- main merge済みで解決済み化できるタスクが `解決済み` へ更新される。
- 対象タスクが `解決済み/` フォルダへ移動され、一覧のリンクも更新される。
- main merge後に関連TSKの解決済み化要否を確認する運用ゲートが明文化される。
- 今後、解決済みタスクが `確認待ち` に滞留するリスクが低減される。
