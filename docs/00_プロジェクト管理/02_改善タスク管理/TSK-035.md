# TSK-035 AI_REPO_RESULT証跡チェック導線と自動化を整備する

- 区分: 運用 / 自動化 / 証跡管理 / docs
- 件名: AI_REPO_RESULT証跡チェック導線と自動化を整備する
- 状態: 未解決
- 優先度: 高
- 実施主体: ChatGPT(会話内) → ChatGPT(リポジトリ編集) / Codex
- 関連箇所: `scripts/check_ai_repo_result.py`, `scripts/test_ai_repo_result.py`, AIリポジトリ作業証跡管理ルール, ChatGPTリポジトリ編集運用ルール, 改善タスク管理ルール, GitHub Actions

## 発端
TSK-032 により、AI_REPO_RESULT証跡を静的検査する `scripts/check_ai_repo_result.py` と回帰テスト `scripts/test_ai_repo_result.py` が追加された。

また、利用手順と正常証跡サンプルも追加され、ユーザーのローカル確認でサンプル証跡PASSと回帰テスト9/9 PASSが確認された。

一方で、TSK-032の作業中に以下の残課題が確認された。

- `AIリポジトリ作業証跡管理ルール.md` へのチェッカー実行導線追記が未対応である。
- PR本文作成後に必ずチェッカーを実行する制度化が未対応である。
- 解決済み化専用PR作成時のチェック導線が未対応である。
- ChatGPT(リポジトリ編集)のPR作成フローへの明文化が未対応である。
- 将来的なCI自動チェックが未対応である。

## 目的
TSK-032で追加した証跡チェッカーを、単体ツールではなくPR本文作成・コミット証跡作成・解決済み化専用PR作成フローの標準ゲートとして定着させる。

あわせて、運用正本への追記、PR作成フローへの制度化、解決済み化専用PRへの適用、将来的なCI化を整理・実装する。

## 対応項目

### 1. AIリポジトリ作業証跡管理ルールへの追記

`AIリポジトリ作業証跡管理ルール.md` に、以下を追記する。

- PR本文作成後は、原則として `scripts/check_ai_repo_result.py` を実行すること。
- `ChatGPT-Repo-Result` / `Codex-Result` の短縮証跡を使う場合は `--format short` を利用できること。
- `unverified` 欠落を意図的に許容する場合は `--allow-missing-unverified` を利用できること。
- チェッカー本体または証跡ルールを変更した場合は `scripts/test_ai_repo_result.py` を実行すること。
- `FAIL` が出た場合は、原則としてPR作成またはmerge前に証跡を修正すること。

### 2. PR本文作成後チェックの制度化

ChatGPT(リポジトリ編集) がPR本文を作成または更新した場合、以下を行う運用を明文化する。

- PR本文を一時ファイル化できる場合はチェッカーを実行する。
- チェッカー実行結果が `PASS` であることを確認してから、確認待ち・merge判断へ進む。
- チェッカーを実行できない場合は、PR本文の `unverified` または作業報告に未実行理由を記録する。

### 3. 解決済み化専用PRへの適用

解決済み化専用PRにおいても、PR本文に `AI_REPO_RESULT` ブロックを残す場合は、証跡チェック対象とする。

検討項目:

- 解決済み化専用PRのPR本文作成後にチェッカー実行を必須化するか。
- 文書・台帳のみの変更で `unverified` がない場合の記録形式を統一するか。
- 自律merge前の条件確認に、証跡チェックPASSまたは未実行理由記録を含めるか。

### 4. ChatGPT(リポジトリ編集)のPR作成フローへの明文化

`ChatGPTリポジトリ編集運用ルール.md` または関連運用規程へ、以下を追記する。

- PR本文作成後に証跡チェッカーを利用する。
- PR本文更新後にも必要に応じて再チェックする。
- 証跡チェック結果をPR本文または作業報告に記録する。

### 5. CI自動チェックの検討または実装

GitHub Actions等で、以下のような自動チェックを実施できるか検討する。

- `scripts/check_ai_repo_result.py` 変更時に `scripts/test_ai_repo_result.py` を実行する。
- PR本文そのものをGitHub Actionsで取得し、`AI_REPO_RESULT` ブロックを検査できるか確認する。
- PR本文取得に必要な権限、フォークPR、GITHUB_TOKEN制約、誤検知リスクを整理する。
- 初回は回帰テストCIのみ、PR本文検査は検討文書化に留める選択肢も含める。

## 完了条件

- GitHub Actions等による自動チェックの要否・範囲・実装可否が整理される。
