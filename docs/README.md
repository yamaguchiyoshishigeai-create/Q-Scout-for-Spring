# Docs Guide

`docs` 配下の文書は、現行の正本、補助仕様、履歴資料、未作成プレースホルダ文書が混在しないように、以下の方針で整理しています。

## docs 直下の基本方針

`docs/` 直下は、原則として docs 全体の入口文書と大分類フォルダを置く層として扱います。

置き場所に迷ったことを理由に、規程文書や運用文書を `docs/` 直下へ追加しないことを原則とします。管理系文書と横断運用規程の現行主軸は `00_プロジェクト管理/` とし、docs 全体または複数工程に横断して適用する正式な運用規程は、まず `00_プロジェクト管理/05_横断運用規程/` への配置を検討してください。

## 大分類フォルダの位置づけ

- `00_プロジェクト管理/`
  - プロジェクト全体または docs 全体に横断的に効く管理文書、運用文書、規程文書の親フォルダであり、現行の管理系主配置先です。
  - 管理系文書の追加は、原則として docs 直下ではなくこの配下で受けます。
  - 下位分類として、`01_プロジェクト計画/`、`05_横断運用規程/` などを持てる構成として扱います。
- `10_企画/`
  - プロジェクト背景、目的、提供価値、将来構想の起点となる企画文書群の配置先です。
- `20_要件定義/`
  - Q-Scout for Spring が何を実現するか、何を対象とするか、どの制約下で成立するかを定義する層です。
- `30_基本設計/`
  - 要件を受けて、システム全体をどの構成・責務分担・主要処理方式で実現するかを定義する層です。
- `40_詳細設計/`
  - 基本設計で定めた骨格を、画面差分・個別 UI・個別部品仕様へ落とし込む層です。
- `50_実装/`
  - 実装工程に継続適用する正式文書の配置先です。実装規約、実装標準、実装完了条件、実装工程の正式運用ルール等を将来的に集約します。
- `60_単体テスト/`
  - 単体テスト工程に継続適用する正式文書の配置先です。現時点ではプレースホルダ運用です。
- `70_結合テスト/`
  - 結合テスト工程に継続適用する正式文書の配置先です。現時点ではプレースホルダ運用です。
- `90_アーカイブ/`
  - 履歴資料、旧フェーズ資料、現行正本ではない過去文書の配置先です。

## 現行補助仕様

- 管理系文書の下位分類:
  - `00_プロジェクト管理/01_プロジェクト計画/`
    - プロジェクト計画関連文書の配置先です。現時点ではプレースホルダ文書を配置しています。
  - `00_プロジェクト管理/01_プロジェクト計画/Public-Free版・Private-Pro版分離運用方針.md`
    - Public リポジトリを Free 試供版兼ポートフォリオとして整備し、商用 Pro 版を別 Private リポジトリで管理するための運用方針です。
  - `00_プロジェクト管理/02_改善タスク管理/`
    - 改善項目の状態管理、優先度整理、実施主体の明確化を行う管理系文書の配置先です。
  - `00_プロジェクト管理/02_改善タスク管理/改善タスク管理ルール.md`
    - 改善タスクの運用ルール、採番、確認待ち、解決済み移動 / 再オープン手順を定義する正式ルールです。
  - `00_プロジェクト管理/02_改善タスク管理/改善タスク課題一覧.md`
    - 全改善タスクを俯瞰する一覧正本です。解決済みタスクも一覧に残し、個別ファイルへリンクします。
- docs 全体または複数工程に横断する運用規程:
  - `00_プロジェクト管理/05_横断運用規程/`
    - `00_プロジェクト管理/` 配下で、docs 全体または複数工程に横断して適用する正式な運用規程の集約先です。
    - 今後、docs 作成運用方針書、文書命名規則、`CodexExec.result` 関連運用規程などを整理する場合も、まずここを候補として検討します。
  - `00_プロジェクト管理/05_横断運用規程/実装系文書配置規程.md`
    - `40_詳細設計/44_実装指示書/` と `50_実装/`、および docs 横断規程の住み分けを定義する正式規程です。
  - `00_プロジェクト管理/05_横断運用規程/ChatGPTリポジトリ編集運用ルール.md`
    - ChatGPT(リポジトリ編集) による局所編集、PR 作成、ユーザー確認、改善タスク解決済み化ゲートを定義する正式な横断運用規程です。
  - `00_プロジェクト管理/05_横断運用規程/Codex連携運用ルール.md`
    - ChatGPT が Codex 向け作業指示を作成し、Codex 実行結果を確認する際に参照する入口文書です。関連する Codex プロンプト作成ルールと `AIリポジトリ作業証跡管理ルール.md` への導線を定義します。
  - `00_プロジェクト管理/05_横断運用規程/AIリポジトリ作業証跡管理ルール.md`
    - ChatGPT(リポジトリ編集) と Codex によるリポジトリ変更時の PR 本文 / PR コメント / コミットメッセージ短縮記録、ローカル実行結果ファイルの扱いを定義する正式な横断運用規程です。
  - `00_プロジェクト管理/05_横断運用規程/公開サンプル評価運用方針.md`
    - 公開 Spring サンプルを用いた比較確認の目的、正式分類、中核候補 / 補助候補 / 重量級候補 / 保留候補の扱いを定義する運用方針です。

## フォルダ維持方針

- 正式文書名の空ファイルは置かない
- docs 配下では、単なる技術的保持よりも人間が一覧で理解しやすいことを優先する
- 必要に応じて、正式文書名のプレースホルダ文書を置き、その文書が未作成であることを明記する
- `.gitkeep` は必要時に使ってよいが、docs 配下ではプレースホルダ文書方式を優先する
