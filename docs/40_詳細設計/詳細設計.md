# 詳細設計

## 1. 文書概要

### 1.1 文書目的

本書は、Q-Scout for Spring における `docs/40_詳細設計/` 配下の文書群について、位置づけ、役割分担、読書順、運用方針を明確化するための正式ガイド文書である。

本書の目的は、詳細設計文書群を単に一覧化することではなく、`docs/30_基本設計/基本設計.md` で定義された骨格を受けて、どの文書がどの粒度の詳細事項を担うのか、どの順で参照すべきか、今後どのように整備を進めるのかを明示することにある。

### 1.2 対象範囲

本書の対象範囲は、`docs/40_詳細設計/` 配下の以下を含む。

- 詳細設計正本の将来配置先
- 現行 UI 差分設計文書
- Figma 運用および UI 運用基盤文書
- Codex 向け実装指示書
- 40 系整理計画および JavaDoc 導入方針文書

本書は、個別差分仕様そのものを再記述することを目的としない。各文書の中身を大量転記するのではなく、位置づけと参照導線を明示する。

### 1.3 この文書の位置づけ

本書は、`docs/40_詳細設計/` の入口文書であり、40 系文書群の総論ガイドとする。

`docs/README.md` が docs 全体の入口であるのに対し、本書は 40 系に入った利用者に対して、40 系内部の構造、文書の役割差、目的別の読み順を案内することを役割とする。

### 1.4 関連文書

関連する主要文書は、以下のとおりとする。

- `docs/README.md`
- `docs/20_要件定義/20_要件定義ガイド.md`
- `docs/30_基本設計/基本設計.md`
- `docs/40_詳細設計/41_詳細設計正本/詳細設計正本.md`

## 2. 40_詳細設計 の基本的位置づけ

### 2.1 40_詳細設計 が担う役割

`40_詳細設計/` は、`30_基本設計/` で定義されたシステム全体の構成、責務分担、主要処理方式を受けて、それらを個別 UI、個別部品、個別差分、運用差分、将来のコード近傍責務文書へ展開する層である。

すなわち 40 系は、「システム全体の骨格」を定義する場ではなく、「骨格を各具体単位へ落とし込む場」である。

### 2.2 30_基本設計 との境界

`30_基本設計/` は、要件を受けてシステム全体をどの構成・責務分担・主要処理方式で実現するかを定義する層である。

一方 `40_詳細設計/` は、その骨格を前提として、以下のような詳細事項を扱う。

- トップページや補助ページの個別 UI 差分
- 表示構造、見た目、レイアウト、導線の詳細
- Figma を用いた確認・修正・反映の運用
- 実装時に必要な差分実装指示
- 将来の JavaDoc 由来責務文書の配置

したがって 40 系では、システム全体骨格を再定義しない。全体構成、提供形態の基本方針、共通解析コアの責務分担などは 30 系を正本とする。

### 2.3 20_要件定義 との境界

`20_要件定義/` は、Q-Scout for Spring が何を実現し、何を対象とし、どの制約下で成立するかを定義する層である。

40 系では、MVP の成立条件、アップロード制約、タイムアウト制約、対象外事項などを新たに定義し直さない。これらの制約は 20 系を前提とし、その制約のもとで画面や運用をどう具体化するかを 40 系で扱う。

### 2.4 実装指示書・運用手順との違い

40 系には詳細設計本文だけでなく、運用手順書や実装指示書も含まれる。ただし、それらは詳細設計正本そのものではない。

- 実装指示書は、設計に基づいて Codex 等へ作業を渡すための実務文書である
- 運用手順書は、Figma 利用や差分確認を回すための実務文書である
- 詳細設計本文は、個別差分や責務を定義する設計文書である

本書では、これらを明確に区別して扱う。

## 3. 40_詳細設計 全体構成

### 3.1 下位フォルダ構成の概要

現在の `docs/40_詳細設計/` は、以下の五分割構成を採用する。

- `41_詳細設計正本/`
- `42_UI差分設計/`
- `43_UI運用・Figma/`
- `44_実装指示書/`
- `45_整理方針・補助文書/`

この構成は、現在の混在状態を整理すると同時に、将来の JavaDoc 由来文書群を主筋として受け入れるための器を先行確保する思想に基づく。

### 3.2 41_詳細設計正本 の役割

`41_詳細設計正本/` は、将来的に JavaDoc 由来またはそれに準ずる形で整備される、詳細設計の主筋を格納するフォルダである。

ここには今後、以下のような文書が入ることを想定する。

- コード近傍の責務文書
- 主要クラス責務文書
- 主要サービス境界文書
- DTO の意味構造文書
- 主要フロー責務文書

現時点では導入準備段階であり、入口文書のみを配置している。

### 3.3 42_UI差分設計 の役割

`42_UI差分設計/` は、現行 UI 差分設計の本文群を格納するフォルダである。

ここには、トップページや関連 UI の再構成、直感訴求強化、レイアウト最適化、成果物サンプル統合など、段階差分として積み上げられてきた設計文書を集約する。

現時点で、40 系本文の実質的な中心はこのフォルダにある。

### 3.4 43_UI運用・Figma の役割

`43_UI運用・Figma/` は、Figma 運用、静的資産外部化、UI 差分反映フローなど、UI の確認・修正・運用を支える基盤文書を格納するフォルダである。

ここには、設計本文だけでなく、ユーザーや作業者が実際に UI 改善を回すための運用文書も含む。

### 3.5 44_実装指示書 の役割

`44_実装指示書/` は、Codex 等へ渡す差分実装プロンプト、作業単位の実行指示書を格納するフォルダである。

ここに置く文書は、設計正本ではなく、設計を具体的な実装作業へ落とし込むための作業用文書として扱う。

### 3.6 45_整理方針・補助文書 の役割

`45_整理方針・補助文書/` は、40 系整理の進め方や JavaDoc 導入方針など、40 系そのものを支える補助方針文書を格納するフォルダである。

ここに置く文書は、詳細設計本文ではないが、40 系をどのように読ませ、どのように将来拡張するかを決める基準文書として重要である。

## 4. 読書順ガイド

### 4.1 初めて 40 系を読む場合

40 系を初めて読む場合は、まず 40 系の整理思想を把握する必要がある。そのため、次の順で読むことを推奨する。

1. `45_整理方針・補助文書/40_詳細設計整理フェーズ実行計画.md`
2. `43_UI運用・Figma/Figma反映運用を含む静的資産外部化仕様書_v1.0.md`
3. `42_UI差分設計/デザイン差分修正仕様書_v1.6.md`
4. 必要に応じて `42_UI差分設計/` 内の v1.1〜v1.4 を遡る
5. `43_UI運用・Figma/ユーザー向け_Figma運用作業手順書_v1.2.md`
6. `45_整理方針・補助文書/JavaDoc導入方針書.md`

### 4.2 現行 UI 差分を把握したい場合

現行のトップページ UI 差分や表示構造の到達点を知りたい場合は、まず `42_UI差分設計/デザイン差分修正仕様書_v1.6.md` を読むことを推奨する。

その後、必要に応じて以下を遡る。

- `左サイドバー前提トップページ再構成_デザイン仕様書_v1.1.md`
- `左サイドバー前提トップページ再構成_差分デザイン仕様書_v1.2.md`
- `トップページ直感訴求強化_差分デザイン仕様書_v1.3.md`
- `トップページ_introレイアウト最適化_差分デザイン仕様書_v1.4.md`

### 4.3 Figma 運用を確認したい場合

Figma による UI 差分確認、スクリーンショット取得、差分作成、実装反映の流れを確認したい場合は、次の順を推奨する。

1. `43_UI運用・Figma/Figma反映運用を含む静的資産外部化仕様書_v1.0.md`
2. `43_UI運用・Figma/ユーザー向け_Figma運用作業手順書_v1.2.md`

前者が基盤方針、後者が実作業手順である。

### 4.4 Codex 実装指示書を探したい場合

Codex へ直接渡す作業文書を探したい場合は、`44_実装指示書/` を参照する。

ただし、ここにある文書は設計正本ではなく、実務上の実装依頼文書であるため、必要に応じて先に `42_UI差分設計/` の設計本文を確認することが望ましい。

### 4.5 JavaDoc 導入方針を確認したい場合

JavaDoc の役割、対象範囲、記述標準、将来の詳細設計正本との関係を確認したい場合は、`45_整理方針・補助文書/JavaDoc導入方針書.md` を参照する。

## 6. 版管理と差分文書の読み方

### 6.1 v1.1〜v1.6 差分文書群の位置づけ

`42_UI差分設計/` にある v1.1〜v1.6 の文書群は、トップページおよび関連 UI の改善を段階的に積み上げてきた差分設計系列である。

これらは互いに無関係な文書ではなく、前段の差分を受けて次段の差分が加わる形で構成されている。

### 6.2 差分文書を時系列で追う場合の見方

経緯を理解したい場合は、v1.1 から順に読むことで、再構成、成果物サンプル強化、直感訴求強化、intro レイアウト最適化、最終的な描画ロジック統合へ至る流れを追うことができる。

### 6.3 現行到達点を優先して読む原則

一方で、現行の判断や設計意図だけを素早く把握したい場合は、もっとも後続版を優先して読む。

すなわち、現在は v1.6 を起点にし、必要な場合のみ過去版を遡る読み方を原則とする。

### 6.4 実装指示書を設計正本と混同しない原則

`44_実装指示書/` にある文書は、設計本文を置き換えるものではない。

設計意図や責務判断を確認する際は、必ず `42_UI差分設計/` および `45_整理方針・補助文書/` を優先し、実装指示書は作業文書として扱う。

## 7. 41_詳細設計正本 の将来方針

### 7.1 41 を先頭に確保している理由

`41_詳細設計正本/` を先頭に確保しているのは、将来的に JavaDoc 由来またはそれに準ずる詳細設計文書群が、詳細設計の主筋になることを見越しているためである。

差分設計や運用文書よりも、本来の詳細責務文書を主筋に据えるため、番号上も最優先位置を先に確保している。

### 7.2 JavaDoc 導入との関係

JavaDoc は、企画・要件・基本設計を置き換えるものではなく、コード責務の補助正本として導入する方針である。

今後、優先対象クラスや主要 DTO に対する JavaDoc 整備が進んだ場合、その成果を整理し、必要に応じて 41 配下へ責務文書として配置する。

### 7.3 今後ここに入る想定文書

将来 41 に格納する可能性がある文書例は、以下のとおりである。

- 主要サービス責務文書
- 主要クラス責務一覧
- 主要 DTO 意味構造文書
- 主要フロー責務文書
- JavaDoc 由来補助文書の正規化版

### 7.4 現時点で未整備であることの明記

現時点では、41 は将来置き場であり、詳細設計正本群の本格整備は未着手または準備段階である。

したがって、40 系利用者は、現時点では 42〜45 を参照し、41 は今後の主筋化に備えた配置先として理解する。

## 8. 40_詳細設計 の運用ルール

### 8.1 新規文書をどこへ置くか

40 系へ新規文書を追加する際は、まずその役割を判定し、以下のいずれかへ置く。

- 詳細設計正本候補 → `41_詳細設計正本/`
- UI 差分設計本文 → `42_UI差分設計/`
- UI 運用・Figma 関連 → `43_UI運用・Figma/`
- 実装依頼・差分実行指示 → `44_実装指示書/`
- 整理方針・導入方針 → `45_整理方針・補助文書/`

### 8.2 差分設計・運用手順・実装指示書の振り分け基準

判断基準は次のとおりとする。

- 設計判断や差分仕様を定義する文書は 42
- 見た目確認、Figma、静的資産管理、手順を扱う文書は 43
- Codex 等へ直接渡す作業指示は 44
- 40 系全体の整理や将来導入方針を扱う文書は 45

### 8.3 役割が曖昧な文書を追加しない原則

どこに置くべきか曖昧な文書を、直下へ雑に追加しないことを原則とする。

役割が曖昧な場合は、まず文書の主目的を明確化し、その上で適切な分類先を決める。必要であれば、本書または `docs/README.md` の導線も更新する。

### 8.4 docs/README との整合維持

40 系構造を変更した場合は、必ず `docs/README.md` 側の案内も更新し、docs 全体入口との整合を保つ。

40 系内部だけ整っていても、docs 全体導線が古いままでは読者に誤解を与えるためである。

### 8.5 将来の統合・アーカイブ判断

差分設計が増えすぎた場合、または 41 の主筋文書が十分整った場合は、古い差分文書を補助設計または履歴資料として整理し直すことを検討する。

ただし現時点では、v1.1〜v1.6 の差分群は現行 UI 改善経緯として参照価値が高いため、即時アーカイブ前提とはしない。

その際は、JavaDoc をそのまま転載するのではなく、必要な観点に整理した上で 41 の主筋文書群へ落とし込む。

### 9.4 全層通読整合レビューへの接続

40 系の構造整理が一定水準に達した後は、`10_企画 → 20_要件定義 → 30_基本設計 → 40_詳細設計` の順で全層通読整合レビューを実施する。

本書は、その際の 40 系入口文書として機能することを想定する。
