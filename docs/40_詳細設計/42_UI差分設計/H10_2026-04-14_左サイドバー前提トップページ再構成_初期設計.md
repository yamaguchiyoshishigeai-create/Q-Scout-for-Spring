# 左サイドバー前提トップページ再構成 初期設計

## 位置づけ

本書は、2026-04-14 に、左サイドバー前提トップページ再構成の初期設計方針を詳細設計として残すために作成された文書である。

## 1. 文書目的
本仕様書は、Q-Scout for Spring の現行 Web UI を、左サイドバー前提の情報設計へ再構成するための設計指針を定義するものである。

目的は以下の3点である。

1. 現行の分散した導線を整理し、アプリ全体の構造を一目で分かるようにする  
2. トップページを単なるアップロード画面ではなく、Q-Scout 全体の入口画面へ昇格させる  
3. 既存の help / preview / 多言語 / 解析結果表示資産を破棄せずに活用する

---

## 2. 現行UIの整理

### 2.1 現行トップページの構成
現行トップページには、以下が縦積みで配置されている。

- 上部リンクと多言語切替
- hero
- 現在の対応範囲
- 現時点で非対応
- zipアップロードパネル
- 実行結果
- 成果物プレビュー / ダウンロード導線

この構成自体は機能的に成立しているが、**説明・入力・結果・成果物** の役割が同一階層で並んでいるため、アプリ全体の構造としては把握しづらい。

### 2.2 現行 help ページの役割
help ページには以下が整理されている。

- 利用の流れ
- 現在の対応範囲
- 成果物
- 推奨ZIP作成方法
- 非対応事項

したがって、このページは削除対象ではなく、**トップから左サイドバー経由で自然に辿れる補助ページ** として活かすべきである。

### 2.3 現行 preview ページの役割
preview ページは、ユーザー向け Markdown および AI入力 Markdown の中身を確認してからダウンロードできる構成になっている。  
これは成果物の価値を直接見せる重要資産であり、トップに全文埋め込むのではなく、**成果物セクションからプレビューへ送る構成** が適切である。

---

## 3. 再構成の基本方針

### 方針A：左サイドバー必須
トップページには常設の左サイドバーを置く。  
この左サイドバーは、単なるリンク一覧ではなく、**説明 → 実行 → 結果確認 → 成果物確認** の流れを示すナビゲーションとして機能させる。

### 方針B：トップページを母艦化
トップページは、今後の Q-Scout Web UI のハブとする。  
ただし、多画面SPA化は行わず、既存のサーバレンダリング基盤を維持する。

### 方針C：既存ページ資産の温存
以下は削除せず活かす。

- 解析画面のアップロード機能
- 結果カード
- help ページ
- Markdown preview ページ
- locale switcher
- エラーモーダル

### 方針D：トップでは要約と導線を担当させる
トップページに詳細情報を全部載せない。  
詳細説明や全文プレビューは既存ページへ委譲し、トップは **入口・要約・導線** に集中する。

## 4. 新トップページの情報設計

### 4.1 全体構成
```text
ヘッダー
├─ ロゴ/タイトル
├─ 主要リンク
└─ 言語切替

ページ本体
├─ 左サイドバー
└─ 右メインコンテンツ
```

### 4.2 左サイドバー項目
以下を固定項目とする。

1. はじめに
2. 診断を実行
3. 結果サマリ
4. 成果物
5. 使い方・仕様
6. このツールについて

### 4.3 採用理由
前回案の「違反一覧」「ルール解説」までトップに直接持ち込む案は、現行資産との整合ではやや先行しすぎる。  
現時点では、すでに存在する help / preview / result を活かしながら、まずは **入口画面の整理** を優先する方が合理的である。

## 5. 各セクション仕様

### 5.1 はじめに
#### 目的
初見ユーザーに Q-Scout の価値を短時間で伝える。

#### 表示内容
- タイトル
- 一文要約
- このツールで得られるもの
- 小さな成果物サンプル
- 診断を実行する導線

#### 文言の方向性
README 冒頭で整理した内容を短く転用する。  
主軸は以下とする。

- Spring 向け品質診断
- 総合スコア可視化
- 人間向け Markdown
- AI入力 Markdown

#### UI要件
- 現行 hero を基礎にする
- 文章は現行より少し構造化する
- 長文説明は避ける

---

### 5.2 診断を実行
#### 目的
既存のアップロード・実行機能を主導線として維持する。

#### 表示内容
- ファイル選択
- 実行ボタン
- 制限カード
  - zip
  - 20MB
  - 60秒
- ZIP作成の目安
- エラー表示
- 実行中表示

#### UI要件
- 現行 form をそのままベースにする
- 現在の対応範囲や非対応一覧をこの近辺へ圧縮再配置してよい
- 非対応の全文常設表示は避ける

---

### 5.3 結果サマリ
#### 目的
解析結果の要約を、アップロードフォームとは独立した意味のある面として見せる。

#### 表示内容
- 対象ファイル
- 実行時刻
- 総合スコア
- 違反件数
- HIGH / MEDIUM / LOW
- 簡潔な所見
- 成果物への導線

#### UI要件
- 現行 result ブロックと metrics を活かす
- アップロード直下の“付随結果”ではなく、サイドバーからアクセスできる主要セクションへ格上げする

---

### 5.4 成果物
#### 目的
Q-Scout の出力物構造をユーザーに明確に理解させる。

#### 表示内容
- ユーザー向け Markdown の説明
- AI入力 Markdown の説明
- 各プレビューへの導線
- 各ダウンロードへの導線

#### UI要件
- 現行 artifact-card を再利用する
- preview ページへのリンクは維持する
- preview ページの全文はトップへ埋め込まない

---

### 5.5 使い方・仕様
#### 目的
help ページへの導線を格上げし、全体の理解負荷を下げる。

#### 表示内容
- トップ内では要約のみ
- 「詳しく見る」で help ページへ遷移

#### UI要件
- 現行 help ページは再利用する
- トップ上では「利用の流れ」「対応範囲」「推奨ZIP作成」などを短く要約してもよい
- ただし主役は help ページ側とする

---

### 5.6 このツールについて
#### 目的
README 的な価値訴求をトップ内にも持つ。

#### 表示内容
- 対象ユーザー
- 技術的見どころ
- CLI資産活用型であること
- Web MVP / 多言語 / AI非依存コアの特徴

#### UI要件
- README 冒頭改善内容と語彙を合わせる
- 過度に長文化しない
- 2〜4ブロック程度の簡潔な情報に留める

---

## 6. レイアウト仕様

### 6.1 ヘッダー
- 現行 topbar を拡張利用
- 左：タイトル/主要リンク
- 右：locale switcher

### 6.2 ページ本体
- 2カラム
- 左：固定または準固定サイドバー
- 右：メインコンテンツ

### 6.3 サイドバー
- デスクトップでは縦固定に近い見え方
- モバイルでは上部折りたたみまたは横スクロール可能な軽量ナビでも可
- 現時点で複雑なアニメーションは不要

### 6.4 メイン
- 現行 card/panel/section のトーンを踏襲
- 角丸・背景・線・アクセントカラーは現行デザインと整合させる
- トップの世界観は維持する

---

## 7. モバイル対応方針
現行UIは既に小画面用の簡易対応がある。  
再構成後も、以下を守る。

- 2カラムをそのまま固定しない
- モバイルではサイドバーを上部ナビ化してもよい
- フォーム、結果カード、成果物カードは縦積み維持
- 可読性を優先し、情報量を過剰に詰め込まない

---

## 8. 多言語対応方針
既存の locale switcher は維持する。  

### 多言語化対象
- サイドバー項目名
- 各見出し
- 補助説明
- エラーメッセージ
- 結果サマリラベル
- ダウンロード文言

### 多言語化しないもの
- 数値
- ファイルパス
- コード断片
- AI入力Markdown本文（原則英語維持）

---

## 9. 今回やらないこと
今回は以下を対象外とする。

- help ページの全面刷新
- preview ページの全面刷新
- ルール解説専用画面の新設
- 詳細違反一覧専用画面の新設
- 履歴管理
- 推移グラフ
- SPA化
- 重いJS導入

---

## 10. 完了条件
以下を満たしたら本仕様の実装完了とする。

1. トップページに左サイドバーが追加されている  
2. トップページが「説明・実行・結果・成果物」のハブとして機能している  
3. 現行の help / preview / locale switcher が自然に統合されている  
4. 現行UIのトーンを壊していない  
5. モバイルでも破綻しない  
6. 多言語切替の導線を壊していない
