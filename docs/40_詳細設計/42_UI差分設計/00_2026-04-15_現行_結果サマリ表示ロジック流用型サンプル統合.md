# 結果サマリ表示ロジック流用型サンプル統合

## 位置づけ

本書は、2026-04-15 に、トップページ成果物サンプル表示を結果サマリ表示ロジック流用型へ再設計する意図を詳細設計として残すために作成された文書である。

## 提案名
**結果サマリ表示ロジック流用型サンプル統合**

## 1. 文書目的
本差分仕様書は、トップページ内の「成果物サンプル」表示について、現在の専用サンプルUI実装をやめ、**実際の結果サマリ表示ロジックをそのまま流用して、サンプルデータで描画する方式** へ変更するための追加仕様を定義するものである。

目的は以下の3点である。

1. サンプル表示と実際の結果サマリ表示の見た目差分を構造的になくす  
2. 今後の UI 変更時に、本番結果サマリを変更すればサンプル側も自動追従する状態を作る  
3. Q-Scout の既存方針である「共通化・再利用・Web層の薄化」に沿った構成へ整理する  

---

## 2. 差分の背景
これまでの差分修正により、トップページ内の成果物サンプルは大幅に改善され、実際の結果サマリにもかなり寄せられた。  
しかし現状では、依然として

- 実際の結果サマリ
- トップ内のサンプル表示

が別系統の表示ロジックとして存在しており、今後 UI 変更が続くたびに **サンプル側と本番側の調整を二重に行う必要** がある。

この状態は、Q-Scout の既存方針である

- CLI資産の最大流用
- 共通解析サービスの活用
- Web層は薄く保つ

という設計思想と整合しにくい。  
そのため、v1.6 では **サンプルも本番結果サマリと同じ描画ロジックを流用し、データだけを sample data に差し替える構成** へ変更する。

---

## 3. 差分の基本方針

### 方針A：結果サマリ表示の描画部品を共通化する
実結果サマリとサンプル結果サマリは、**同じマークアップ構造 / 同じスタイル系統 / 同じ描画部品** を使う。

### 方針B：サンプル専用UIを廃止する
トップ内のサンプルを見せるためだけの専用UI実装は持たない。  
見せ方は本番結果サマリと同一にし、サンプルでは sample data を流し込む。

### 方針C：サンプルであることは明示する
見た目を共通化しても、トップに表示するものは sample 表示であるため、補助文や見出しでサンプルであることを明確にする。

### 方針D：実データ表示と sample 表示の責務を分ける
本番結果は従来どおり `response` 等の実データで表示する。  
トップ内の sample は、固定の sample data を用いて描画する。

---

## 4. 差分対象

### 主対象
- `src/main/resources/templates/index.html`
- 結果サマリ表示部
- トップページ内の成果物サンプル表示部

### 補助対象
- 必要に応じて ViewModel / DTO / Controller
- `messages_ja.properties`
- `messages_en.properties`
- 必要なら表示確認テスト

### 原則非対象
- 解析ロジック本体
- スコア計算ロジック
- help ページ
- preview ページ
- 左サイドバー構造
- 全体のページレイアウト

---

## 5. 追加仕様

## 5.1 結果サマリ表示部の共通化
実結果サマリ表示とトップ内サンプル表示で、同一の描画部品を使うようにする。

### 推奨方式
- Thymeleaf fragment 化
- または同一テンプレート内での共通 partial 化
- または共通ブロック化

### 要件
- 実表示とサンプル表示でマークアップ差分を最小化する
- 今後のUI変更時に1か所修正で済むこと
- 表示内容の責務が読みやすいこと

---

## 5.2 サンプル用データの導入
トップページ内のサンプル表示用として、固定の sample data を導入する。

### サンプル用データに含める項目
- 完了メッセージ相当
- 対象ファイル名
- 疑似実行時刻
- 総合スコア
- 違反件数
- HIGH / MEDIUM / LOW
- サンプル側成果物導線用情報
- サンプル表示フラグ

### 要件
- サンプルは実解析に依存せず描画できること
- 実データとの混線を起こさないこと
- 既存の結果サマリ表示が想定している情報構造と整合すること

---

## 5.3 トップ内サンプル表示で共通結果サマリを使用する
トップページの成果物サンプル領域では、専用の見た目を持たず、共通結果サマリ表示部に sample data を渡して描画する。

### 要件
- 見た目は実際の結果サマリと同じ構造になる
- サンプル用補助見出し・補助文で「サンプル」であると分かるようにする
- 解析前でも表示できること

---

## 5.4 実結果表示と sample 表示の分岐整理
表示分岐は以下のように明確化する。

### 推奨例
- 実データがある場合：実結果サマリを表示
- トップ内の成果物サンプル：常に sample data を用いた共通結果サマリを表示

または

- 共通 fragment に `summaryModel` と `isSample` を渡して描画する

### 要件
- `response == null` / `response != null` の既存挙動を壊さない
- 条件分岐が過剰に複雑化しない
- テンプレートの可読性が維持される

---

## 5.5 サンプル用導線の整理
共通表示を使う場合、本番用の preview / download 直リンクをそのままサンプルへ流すわけにはいかない場合がある。  
そのため、sample 表示時にはサンプル用導線へ適切に置き換える。

### 推奨
- 解析前のサンプルでは `#artifacts` など適切な導線へつなぐ
- 実表示では従来どおり preview / download を使う
- もしくは sample 用 preview ページがあるならそこへ導線する

### 要件
- 実表示の導線は壊さない
- サンプル導線が不自然でない
- 解析前後で意味が破綻しない

---

## 5.6 サンプルであることの明示
サンプル表示は本番結果サマリと同じ見た目を使うため、誤認防止が必要である。

### 方式例
- 見出しに「サンプル」「見本」を付ける
- 補助文を追加する
- トップの成果物サンプル領域名で sample と分かるようにする

### 要件
- 見た目は共通でよい
- ただし本物の解析結果と誤解されないこと

---

## 5.7 多言語対応方針
追加する sample 用補助文やラベルがある場合は、既存の MessageSource 構造へ追加する。

### 要件
- 日本語 / 英語両対応
- 既存命名規則に合わせる
- ハードコードを増やしすぎない

---

## 5.8 テスト方針
必要に応じて、トップページと結果サマリ表示確認を最小限更新する。

### 最低限確認したいこと
1. トップで sample data を用いた結果サマリが表示される  
2. 実結果サマリも従来どおり表示される  
3. 日本語 / 英語で描画できる  
4. `response == null` / `response != null` を壊していない  

---

## 6. 今回やらないこと
- 解析ロジックの変更
- スコア計算ロジックの変更
- help ページ改修
- preview ページ改修
- 新規グラフライブラリ導入
- SPA化
- 履歴管理追加
- ページ全体構造の再設計

---

## 7. 完了条件
以下を満たしたら v1.6 差分実装完了とする。

1. トップの成果物サンプルが、実際の結果サマリ表示ロジックを流用している  
2. サンプル用データ差し替えで表示できる  
3. 実際の結果サマリ変更がサンプルにも自然に反映される構造になっている  
4. 実表示と sample 表示の責務分離が明確  
5. 多言語対応を壊していない  
6. 既存の表示機能を壊していない
