# Figma反映運用を含む静的資産外部化仕様書 v1.0

## 1. 文書概要

### 文書名
Figma反映運用を含む静的資産外部化仕様書

### 提案名
静的資産外部化＋Figma反映運用基盤整備方針 v1.0

### 目的
本仕様書は、Q-Scout for Spring の Web UI において、HTML テンプレート内へ内包されている CSS / JavaScript を外部ファイルへ分離し、あわせて Figma を用いた UI 確認・修正・反映が円滑に行える運用基盤を定義するものである。

本仕様の目的は以下の4点である。

1. テンプレート責務を明確化し、保守性と差分レビュー容易性を高める
2. 将来の UI 改善サイクルに耐えられる静的資産構造を整える
3. 多言語化済み UI と説明系導線を壊さずに見た目改善を継続可能にする
4. ユーザーが Figma 上で視覚的に確認・修正した内容を、実装へ低摩擦で反映できる体制を作る

---

## 2. 背景

### 2.1 現状
現状の改善課題として、トップページの成果物サンプル追加、価値訴求文追加、README再設計などが進行している。
この流れの中で、index.html 内にある CSS / JavaScript を外部ファイルへ分離することが次の主要課題として位置付けられている。

### 2.2 課題認識
現状のままインライン CSS / JavaScript をテンプレート内に保持し続けると、以下の問題が生じやすい。

1. HTML テンプレートが肥大化し、構造把握が難しくなる
2. UI 改善のたびに ChatGPT / Codex へ文章で抽象的に伝える必要があり、伝達ロスが発生しやすい
3. デザイン差分の管理単位が曖昧になり、修正影響範囲を把握しにくい
4. 多言語化された文言層と見た目層が混ざりやすく、責務分離が崩れる
5. 将来的に Figma ベースで UI を確認・修正したい場合、実装反映先が整理されておらず、差分反映コストが高い

### 2.3 今回の判断
本課題は単なる「index.html の整理」ではなく、Web UI 全体の静的資産管理方針を整備し、Figma を UI 仕様の実務上の一次ソースとして活用しやすくするための基盤改善として扱う。

## 3. 基本結論

### 提案名
静的資産外部化＋Figma差分反映運用方式

### 優先度
最優先

### 結論
本仕様では、以下を採用する。

1. HTML テンプレートから CSS / JavaScript を原則外部ファイルへ分離する
2. テンプレートは「構造」「文言表示」「Thymeleaf 式」に責務を集中させる
3. CSS / JavaScript は共通資産とページ固有資産に分離する
4. Figma は「実装コード自動生成ツール」ではなく、「UI差分確認と視覚的仕様調整の基準資産」として扱う
5. Figma の変更内容を実装へ反映する際の運用フローと責務分担を明文化する
6. 既存の Web / i18n / preview / download / CLI 資産は壊さない

## 4. スコープ

## 4.1 対象
本仕様の対象は以下とする。

### A. Web UI テンプレート
- トップページ
- help ページ
- preview ページ
- ルール詳細解説ページ
- その他ユーザー向けテンプレート

### B. 静的資産
- CSS
- JavaScript
- 画像 / アイコン等の UI 補助資産

### C. UI変更運用
- Figma での確認
- Figma での差分修正
- 実装反映手順
- レビュー手順

## 4.2 非対象
本仕様では以下は原則対象外とする。

1. Figma からの完全自動コード生成
2. Figma プラグインとリポジトリの双方向自動同期
3. デザインシステムの大規模再構築
4. SPA化
5. JavaScript フレームワーク導入
6. UI以外の解析コア変更
7. CLI の UI 改修
8. AI 用 Markdown 生成ロジックの変更

## 5. 設計原則

### 5.1 テンプレート責務集中原則
HTML テンプレートは以下に責務を集中させる。

- 画面構造
- セマンティックな領域分割
- MessageSource キー参照
- Thymeleaf 条件分岐
- データの表示位置定義

以下はテンプレートに原則残さない。

- 大量のスタイル定義
- 汎用 JavaScript ロジック
- 長大なページ制御コード

### 5.2 静的資産分離原則
見た目と振る舞いは外部ファイルへ分離する。

- CSS は static/css 配下
- JavaScript は static/js 配下
- 必要に応じてページ固有ファイルを追加する

### 5.3 薄い改善優先原則
MVP改善フェーズであることを踏まえ、過設計を避ける。

- 最初から複雑なデザインシステムを導入しない
- まずは共通ファイル＋必要最小限のページ別ファイルでよい
- 既存UIを壊さず、静的資産の責務整理を優先する

### 5.4 Figma一次仕様補助原則
Figma は UI 差分の視覚的合意形成を担う。
ただし、最終的な動作仕様・DOM構造・多言語化・アクセシビリティ整合は実装側で担保する。

### 5.5 多言語化保全原則
Figma で見た見た目差分を反映する際も、以下を守る。

- 文言ハードコードを増やさない
- 表示文言は MessageSource で解決する
- 日本語 / 英語のレイアウト破綻を確認する
- 説明系 UI と多言語化の整合を維持する

## 6. 採用方針

## 6.1 静的資産の基本構成
以下の構成を推奨する。

src/main/resources/
  templates/
    index.html
    help.html
    preview.html
    rule-help.html
    ...
  static/
    css/
      app.css
      home.css
      help.css
      preview.css
      rule-help.css
    js/
      app.js
      home.js
    img/
      ...

### 基本方針
- `app.css` / `app.js` は共通資産
- `home.css` / `home.js` 等はページ固有差分
- ページ固有ファイルは必要な場合のみ作成する
- 初回実装では「共通1本＋トップページ用1本」から開始してよい
- 補助ページは `app.css` を基盤とし、必要な差分だけを `help.css` `preview.css` `rule-help.css` へ分離する

### 実装メモ v1.0
- `index.html` は `app.css` + `home.css` + `app.js` + `home.js` を参照する
- `help.html` は `app.css` + `help.css` を参照する
- `preview.html` は `app.css` + `preview.css` を参照する
- `rule-help.html` は `app.css` + `rule-help.css` を参照する
- Figma 差分の実装位置を追いやすくするため、主要セクションに `data-figma-section` 属性を付与する

## 6.2 テンプレート記述方針
HTML テンプレートには以下のみを残す。

1. 画面の DOM 構造
2. セクション / カード / フォーム / 結果領域などの意味的区分
3. `th:text` `th:if` `th:each` 等の Thymeleaf 記述
4. 外部 CSS / JS の参照
5. 必要最小限の data 属性

### 非推奨
- `<style>` ブロックの大量残置
- `<script>` ブロックの長大な残置
- 直接文字列を埋め込んだ JavaScript
- DOM構造へ強く依存した場当たり的セレクタ設計

## 7. CSS 設計方針

## 7.1 レイヤ分け
CSS は以下のレイヤを意識して構成する。

1. ベース
- body
- typography
- spacing
- color tokens 相当の基本値

2. 共通コンポーネント
- button
- card
- section
- alert
- badge
- form group
- result panel

3. ページ固有差分
- home page 専用ヒーロー
- サンプル成果物カード
- preview page 専用表示
- help page 専用説明レイアウト

## 7.2 命名方針
クラス名は意味ベースで整理する。

推奨例：
- `page-shell`
- `section-block`
- `hero-panel`
- `artifact-card`
- `result-summary`
- `download-links`
- `lang-switcher`

避けるべきもの：
- `box1`
- `tmp-area`
- `left-part-new2`
のような意味不明確な命名

## 7.3 Figma反映を意識した方針
Figma と実装の対応を取りやすくするため、主要コンポーネント単位でクラス名を安定させる。
可能であれば Figma のフレーム / セクション名と実装コンポーネント名を近づける。

---

## 8. JavaScript 設計方針

## 8.1 基本原則
JavaScript は必要最小限とする。

対象例：
- ファイルサイズ事前検知
- モーダル開閉
- 実行中表示の制御
- フォーム送信時の二重送信防止
- 軽量な表示補助

## 8.2 分離方針
- 共通ロジックは `app.js`
- トップページ固有処理は `home.js`
- preview 固有処理があれば `preview.js`

## 8.3 多言語化整合
JavaScript 内へ日本語 / 英語文言を直接埋め込まない。
必要な場合は、data 属性またはテンプレートから受け渡す。

例：
- `data-max-size-message`
- `data-running-label`
- `data-invalid-upload-message`

---

## 9. Figma運用方針

## 9.1 基本位置付け
Figma は以下の目的で使う。

1. ユーザーが UI の完成イメージを直接確認する
2. 余白、配置、情報の優先度を視覚的に調整する
3. テキストでは伝わりにくい差分を具体化する
4. 実装前に見た目の合意を形成する

### 採用しない位置付け
- Figma をそのまま HTML / CSS に変換する単独正解源
- 動的ロジックまで定義する完全仕様書
- MessageSource や Thymeleaf を置き換える運用

## 9.2 Figma管理対象
優先的に Figma 管理対象とする範囲は以下。

1. トップページ
- ファーストビュー
- 成果物サンプル領域
- 診断実行領域
- 結果サマリ領域
- 使い方 / 補助説明導線

2. 補助ページ
- help ページ
- ルール詳細ページの基本レイアウト
- preview ページの表示骨格

3. 共通部品
- ヘッダ
- フッタ
- ボタン
- カード
- メッセージボックス
- 言語切替導線

## 9.3 Figma非管理対象
以下は Figma で厳密管理しなくてよい。

- 解析結果そのものの可変データ内容
- requestId 等の内部識別値
- Markdown本文の完全レンダリング結果
- エンジン内部仕様
- 例外クラス構造

## 10. Figma差分反映フロー

### 提案名
Figma差分確認→実装反映→画面検証フロー

### 基本フロー
1. ユーザーが Figma 上で修正案を作成または更新する
2. 変更対象ページ・対象部品・変更意図を明示する
3. 実装担当が差分を静的資産へ落とし込む
4. テンプレート / MessageSource / CSS / JS のどこを変更すべきかを整理する
5. 実装後に日本語 / 英語両表示で確認する
6. 必要に応じて Figma 側へ反映済み注記を残す
7. UI差分が大きい場合は改善履歴へ記録する

### 差分依頼時の最低限必要情報
- 対象ページ
- 対象セクション名
- 変更前 / 変更後の見た目
- 変更目的
- 優先度
- 文言変更有無
- 多言語影響有無

---

## 11. 実装反映ルール

## 11.1 変更の受け口
Figma 差分を受けた場合、実装先は以下の順で判断する。

1. 見た目のみ変更
- CSS を優先修正

2. DOM構造変更が必要
- テンプレートを修正

3. 文言変更を含む
- MessageSource を修正
- 必要に応じてテンプレートキーを修正

4. UI挙動変更を含む
- JavaScript を修正
- 必要最小限に留める

## 11.2 実装時の禁止事項
- Figma の見た目に合わせるためだけに文言をハードコードしない
- 日本語だけ合えばよいという実装にしない
- その場しのぎの inline style を追加しない
- DOM へ過度に依存した JS を増やさない
- 既存の preview / download / upload フローを破壊しない

## 11.3 例外
小規模な一時検証のために最小限の inline style / inline script を使うことはあり得る。
ただし、本番反映時には外部ファイルへ戻すことを原則とする。

- 実際のテンプレート / CSS / JS / MessageSource 改修
- 動作確認と差分整理
- 必要最小限の構造改善

責務：
- 実装結果と Figma の差分確認
- 採用向け・ポートフォリオ向け見せ方の最終判断

## 18. 最終結論

本仕様の本質は、Q-Scout for Spring の Web UI を
「その場しのぎで都度修正する構造」から、
「静的資産が分離され、Figma を用いた視覚的差分確認にも耐えられる構造」
へ段階的に移行することにある。

すなわち、今回の改善は単なる CSS / JavaScript 外部化ではなく、

- テンプレート責務の整理
- UI改善サイクルの効率化
- 多言語化整合の維持
- Figmaベースの修正反映体制構築

を同時に前進させる基盤整備である。

この方針により、今後の UI 改善は
「ChatGPT や Codex に抽象的な文章だけで依頼する」方式から、
「Figma で具体化された差分を、整理済みの静的資産へ反映する」方式へ移行しやすくなる。

以上を、Figma反映運用を含む静的資産外部化の正式仕様とする。
