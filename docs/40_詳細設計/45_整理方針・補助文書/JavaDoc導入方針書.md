# JavaDoc導入方針書

## 1. 文書概要

### 1.1 文書目的

本書は、Q-Scout for Spring における JavaDoc 導入の目的、位置づけ、適用範囲、記述標準および運用方針を定めるものである。

JavaDoc は、企画、要件定義、基本設計を置き換えるための仕組みではなく、コード責務を実装近傍で明示し、`docs/40_詳細設計/` の一部やコード責務一覧を補助するための文書化手段として導入する。

### 1.2 適用対象

本書は、Java 実装として存在するアプリケーション層、共通解析サービス層、Web 入口周辺、主要 DTO 群の JavaDoc 整備に適用する。

### 1.3 関連文書

関連する主要文書は、以下のとおりとする。

- `docs/10_企画/Q-Scout-for-Spring-プロジェクト企画書-v2.0.md`
- `docs/20_要件定義/20_要件定義ガイド.md`
- `docs/30_基本設計/基本設計.md`
- `docs/40_詳細設計/45_整理方針・補助文書/40_詳細設計整理フェーズ実行計画.md`
- `docs/40_詳細設計/` 配下の詳細設計文書群

## 2. JavaDoc導入の背景

Q-Scout for Spring は、企画、要件定義、基本設計、詳細設計の docs を段階的に整備してきた。一方で、コード上の責務境界、入力値の意味、戻り値の扱い、例外や補助 DTO の役割については、実装を直接読まなければ把握しにくい箇所が残りやすい。

特に `SharedAnalysisService`、`WebAnalysisService`、`CliApplication`、Controller 群、主要 DTO 群は、システムの入口、実行統括、共通解析、画面応答の接点となるため、コード責務を短く正確に示す補助正本があると保守性と読解性が高まる。

そのため本書では、全面的な仕様書自動生成ではなく、コード責務の補助正本として JavaDoc を段階導入する方針を定める。

---

## 3. 導入の基本方針

JavaDoc 導入の基本方針は、以下のとおりとする。

第一に、JavaDoc は `10_企画`、`20_要件定義`、`30_基本設計` の置換ではない。何を実現するか、どの制約下で成立するか、どの骨格で実現するかは引き続き docs 正本文書で管理する。

第二に、JavaDoc はコード責務の補助正本として扱う。すなわち、クラス、メソッド、主要 DTO が何を担うか、どの前提で呼び出されるか、どの成果物や応答へつながるかを、実装近傍で明示するために用いる。

第三に、導入は段階的に進める。いきなり全クラスへ適用せず、詳細設計や責務把握への効果が高い箇所から着手する。

第四に、JavaDoc 記述は `docs/30_基本設計/基本設計.md` と整合しなければならない。JavaDoc だけで独自の構造や責務を定義してはならない。

---

## 4. 文書体系との関係

Q-Scout for Spring の文書体系における JavaDoc の位置づけは、以下のとおりとする。

- `10_企画` は、プロダクトの目的、価値、到達点、将来テーマを示す
- `20_要件定義` は、何を実現するか、何を対象とするか、どの制約下で成立するかを示す
- `30_基本設計` は、システム全体の構成、責務分割、主要処理方式を示す
- `40_詳細設計` は、基本設計の骨格を個別 UI、個別部品、個別フロー、運用差分へ落とし込む
- JavaDoc は、上記文書を補助するコード近傍の責務説明として扱う

したがって JavaDoc は、docs 正本文書を置き換えるものではなく、詳細設計の一部やコード責務一覧を補完する補助正本である。

---

## 5. 導入対象範囲

導入対象範囲は、まずコード責務の把握と docs 連携効果が高い範囲へ限定する。

対象に含むものは、アプリケーションサービス、Web 入口、CLI 入口、主要 DTO、主要成果物 DTO、必要に応じた補助サービスとする。

一方で、全ルール実装、単純 getter/setter 相当、自己説明的な短小 private メソッド、全面的な自動生成成果物は初期導入対象外とする。

---

## 6. 記述対象と記述粒度

記述対象は、クラス、公開メソッド、必要に応じて主要フィールドまたは enum とする。

記述粒度は、実装詳細を逐語的に説明するのではなく、責務、前提、主要入力、主要出力、呼び出し上の注意点が短時間で理解できる水準とする。

クラス JavaDoc では、そのクラスがシステム全体のどこに位置し、何を担い、何を担わないかを明示する。メソッド JavaDoc では、引数、戻り値、例外、成果物、副作用の要点を簡潔に示す。DTO JavaDoc では、インスタンスが表す意味と利用文脈を示す。

---

## 7. JavaDoc記述標準

JavaDoc 記述標準は、以下の方針とする。

クラス JavaDoc では、責務、利用場面、関連する上位文書との整合を意識する。メソッド JavaDoc では、`@param`、`@return`、必要に応じて `@throws` を用い、引数や戻り値の意味が文脈なしでも理解できるようにする。

記述は過度に長文化せず、日本語を基本に、識別子や技術用語はコード上の名称を尊重する。既に英語命名が定着している型名や成果物名はそのまま使用してよい。

また、以下を標準とする。

- 基本設計と矛盾する責務定義を書かない
- 実装されていない振る舞いを既定事項として書かない
- UI 見た目や要件本文の再記述をしない
- 一読して責務が分かる簡潔さを優先する

---

## 8. 優先導入対象

優先導入対象は、以下のとおりとする。

最優先は `SharedAnalysisService`、`WebAnalysisService`、`CliApplication` である。これらは CLI / Web / 共通解析の接点であり、30 系基本設計との対応を示しやすい。

次点で、主要 Controller 群を対象とする。Web 入口が何を受け取り、どこまでを担い、どこから先をサービスへ委ねるかを明確にするためである。

その次に、主要 DTO 群を対象とする。具体的には `AnalysisRequest`、`AnalysisResult`、`ScoreSummary`、`ReportArtifact`、`ExecutionSummary` など、共通解析や表示導線の理解に直結する型を優先する。

---

## 9. 生成物活用方針

JavaDoc 生成物は、まず開発者の読解補助と責務確認に用いる。加えて、`40_詳細設計` の一部文書やコード責務一覧を作成・更新する際の補助資料として活用する。

ただし、生成物をそのまま正式仕様書として配布することは当面の目的としない。生成 HTML や一覧出力は参照補助とし、正式な上位文書体系は引き続き docs 側で管理する。

---

## 10. 運用ルール

運用ルールは、以下のとおりとする。

JavaDoc を追加または更新する際は、対象クラスの責務が `docs/30_基本設計/基本設計.md` と整合していることを確認する。責務変更を伴う場合は、先に設計文書または実装のどちらが正本かを確認し、独断で JavaDoc だけを先行させない。

また、JavaDoc の更新はコード変更と同時に行うことを原則とする。責務が変わったのに JavaDoc が古いまま残る状態を避けるためである。

さらに、詳細設計文書へ転記する際は、JavaDoc の文章をそのまま大量転載せず、必要な観点へ整理して利用する。

---

## 11. 導入ステップ

導入ステップは、以下の順で進める。

まず、40 系整理結果を踏まえて JavaDoc 導入対象候補を確定する。次に、`SharedAnalysisService`、`WebAnalysisService`、`CliApplication` へ試行導入し、記述粒度と標準を固める。その後、Controller 群と主要 DTO 群へ展開する。

並行して、生成物の活用方法を整理し、必要であれば `40_詳細設計` 側へ責務一覧や参照ガイドを追加する。最後に、導入対象拡大の要否をレビューする。

---

## 12. 成功条件

成功条件は、JavaDoc が docs の代替ではなく補助正本として機能していることである。

具体的には、優先導入対象の責務が短時間で把握でき、`docs/30_基本設計/基本設計.md` と矛盾せず、`40_詳細設計` の一部整理や責務一覧作成に実際に活用できる状態を成功とする。

---

## 13. 今後の拡張余地

将来的には、優先導入対象の成果を踏まえ、補助サービス、代表的なルール群、エラー処理方針、成果物生成系の JavaDoc を段階的に拡張する余地がある。

また、責務一覧やクラス関係の補助資料を `40_詳細設計` へ整備し、JavaDoc 生成物との参照関係を明示することも検討余地とする。ただし、全面的な仕様書再生成や自動化を現時点の既定方針とはしない。
