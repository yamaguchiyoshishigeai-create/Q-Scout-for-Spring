# 提案名：ユーザー向け Figma運用作業手順書 v1.2

## 1. この手順書で解決すること

この手順書は、Q-Scout for Spring の現行 Web UI を Figma 上で確認し、ユーザーが見た目を修正し、その差分を ChatGPT / Codex に実装反映してもらうための**実作業手順**を示す。

今回は特に、次の2点を具体化する。

1. Figma を実際にどう開くか
2. 現行システムの UI を Figma 上でどう表示し、どう編集可能な状態へ持ち込むか

---

## 2. 先に結論

### 推奨運用
初回は、以下の順で進める。

1. **Figma はブラウザ版で開く**
2. **現行 UI は Chrome DevTools の full size screenshot 機能でページ全体を1枚の画像として保存する**
3. **その画像を Figma に取り込む**
4. **見た目の修正案を Figma 上に重ねて作る**
5. **必要になった場合のみ、Webページ取り込み系プラグインを使う**

### この順を推奨する理由
- ブラウザ版は導入が軽い
- Chrome の full size screenshot は長いページでも1枚化しやすい
- スクリーンショット取り込みは必ず動く
- プラグイン方式は便利だが、第三者製で、再現度や制約に差がある
- 初回は「確実に回る運用」を優先した方が良い

---

## 3. Figmaはブラウザ版とデスクトップ版のどちらを使うか

### 推奨
**まずはブラウザ版を使う。**

Figma はブラウザで開ける。デスクトップアプリもあるが、まずはブラウザ版で十分である。

### ブラウザ版を推奨する理由
- 追加インストールなしで始めやすい
- ChatGPT との並行作業がしやすい
- 初回実証には十分
- デスクトップアプリ前提の社内設定や権限問題を避けやすい

### デスクトップ版を使ってもよい場合
- すでに日常的に Figma アプリを使っている
- ブラウザよりアプリの方が作業しやすい
- ファイルを常にデスクトップアプリで開きたい

---

## 4. 事前に準備するもの

作業開始前に、以下を用意する。

### 必須
1. Figma アカウント
2. Q-Scout for Spring の現行画面を開ける環境
3. Chrome ブラウザ
4. スクリーンショットを保存できる環境
5. 修正対象画面のURLまたは表示手順メモ

### あると便利
6. Figma の編集権限があるファイル
7. 必要なら Figma プラグイン利用権限

## 5. Figma を開く具体的手順

### 手順A：ブラウザ版で開く
1. ブラウザで Figma にログインする
2. Drafts か対象プロジェクトを開く
3. 新規デザインファイルを作る
4. ファイル名を分かりやすく付ける

### 推奨ファイル名
- `Q-Scout_UI_Review_Home_v1`
- `Q-Scout_UI_Review_Help_v1`
- `Q-Scout_UI_Review_Preview_v1`

### ファイル内の基本構成
1ファイル内に以下のフレームを作ると分かりやすい。

- `Current_UI`
- `Proposed_UI`
- `Notes`

### 手順B：デスクトップアプリで開く場合
1. Figma デスクトップアプリを起動する
2. ログインする
3. 新規デザインファイルを作る
4. 必要ならブラウザで開いたファイルをデスクトップアプリで開く

## 6. 現行システムのUIをFigmaに表示する方法

現行 UI を Figma に持ち込む方法は、実務上は次の2つである。

### 方法1：スクリーンショットを取り込む
**初回はこれを推奨する。**

#### 特徴
- 必ずできる
- すぐ始められる
- 見た目確認には十分
- そのままの画面状態を基準画像にできる
- ただし、そのままではHTMLのような意味で編集可能ではない

#### どんな時に向くか
- 初回実証
- レイアウトや見せ方を検討したいとき
- プラグインを使わずに始めたいとき
- 実装済み画面の改善案を素早く作りたいとき

### 方法2：Webページ取り込み系プラグインを使う
**2回目以降や、編集可能なレイヤーを増やしたい時に検討する。**

#### 特徴
- スクリーンショットより編集しやすい場合がある
- WebページをFigmaレイヤーへ変換できることがある
- ただし第三者製で、再現度や制約がある
- 画面によっては崩れたり、取り込み後の手修正が必要になる

---

## 7. 初回推奨手順：Chrome のページ全体スクリーンショット方式

### STEP 1：現行画面を Chrome で開く
対象ページを実際に Chrome で開く。

例：
- トップページ
- help ページ
- preview ページ
- rule-help ページ

### STEP 2：Chromeでページ全体のスクリーンショットを1枚で取得する
修正対象ページは、通常の画面キャプチャではなく、**Chrome DevTools の full size screenshot 機能**でページ全体を1枚の画像として保存する。

#### Windows / Linux の手順
1. Chrome で対象ページを開く
2. `F12` または `Ctrl + Shift + I` で DevTools を開く
3. `Ctrl + Shift + P` で DevTools のコマンドメニューを開く
4. `screenshot` と入力する
5. `Capture full size screenshot` を選択する
6. ページ全体のスクリーンショット画像が保存される

#### Mac の手順
1. Chrome で対象ページを開く
2. `Option + Command + I` で DevTools を開く
3. `Command + Shift + P` で DevTools のコマンドメニューを開く
4. `screenshot` と入力する
5. `Capture full size screenshot` を選択する
6. ページ全体のスクリーンショット画像が保存される

#### 補足
- この方法を使うと、縦に長いページでも分割せずに1枚で保存しやすい
- 初回の Figma 運用では、この画像を `Current_UI` の基準画像として使う
- 必要に応じて、日本語表示版と英語表示版をそれぞれ保存する
- 画面幅によって見え方が変わるため、撮影時のブラウザ幅は固定した方がよい

### STEP 3：Figmaファイルを開く
前述の手順で Figma の新規ファイルを開く。

### STEP 4：取得したスクリーンショットを Figma に取り込む
取得したページ全体スクリーンショットを、`Current_UI` フレームへ配置する。

#### 具体手順
1. `Current_UI` 用のフレームを作る
2. スクリーンショット画像を Figma のキャンバスへドラッグ＆ドロップする  
   もしくは  
   `Shift + Command + K`（Mac） / `Shift + Ctrl + K`（Windows）で画像配置を使う
3. 画像を配置する
4. 必要ならフレームサイズを画像に合わせる

### STEP 5：編集用の複製を作る
1. 取り込んだ現行UI画像を複製する
2. 複製を `Proposed_UI` 側へ置く
3. こちらを修正案作成用のベースにする

### STEP 6：画像をロックする
誤ってベース画像を動かさないように、現行UI画像はロックする。

### STEP 7：修正案を上に重ねる
画像そのものを編集するのではなく、Figma 上で以下を重ねて差分案を作る。

- テキスト
- 長方形
- 線
- 矢印
- カードの仮置き
- 新しいボタン形状
- 余白変更を示すガイド

#### ここでの考え方
- まずは「何をどう変えたいか」を可視化する
- 最初から完全な再現を目指さない
- 改善案が伝わることを重視する

### STEP 8：変更箇所に番号を振る
例：
- 変更1：成果物サンプルを上へ移動
- 変更2：診断実行ボタンを拡大
- 変更3：説明文を3点に要約
- 変更4：カード間余白を広げる

### STEP 9：Notes フレームへ理由を書く
`Notes` フレームに、変更理由を短く書く。

例：
- 初見ユーザーが価値を即理解できるようにする
- 実行導線が埋もれていたため
- 情報量過多を軽減する
- 英語時の折り返し余裕を確保する

## 8. もう一段編集しやすくしたい場合：プラグイン方式

### 前提
この方法は便利だが、**第三者製プラグイン前提**である。
初回は必須ではない。

### 推奨用途
- スクリーンショットだけでは編集しづらい
- レイヤー単位で触りたい
- 2回目以降の改善効率を上げたい

### 手順の考え方
1. Figma ファイルを開く
2. プラグインを使える状態にする
3. Webページ取り込み系プラグインを起動する
4. 現行ページを取り込む
5. 取り込み結果を手修正する

### 重要な注意
- 取り込み結果はそのまま正解とは限らない
- 複雑な画面や動的画面は崩れることがある
- 未公開画面やローカル画面は取り込み方法に制約が出る場合がある
- 取り込み後に Figma 上で整理し直す前提で考える

### この方法を使うかどうかの判断
次の条件を満たしたら検討してよい。
- スクリーンショット運用を1回回した
- もっと効率化したい
- プラグイン利用が許可されている
- 多少の崩れを手修正する前提でよい

## 9. 現行UIを「編集可能な状態」にするとは何か

ここは誤解しやすいので明確にする。

### スクリーンショット方式
- 背景画像として現行UIを置く
- その上に修正案を重ねる
- **画像自体は直接編集しない**
- ただし、改善意図を伝えるには十分

### プラグイン方式
- レイヤーや要素として取り込める場合がある
- スクリーンショットより編集しやすい
- ただし完全再現ではない
- 取り込み後の掃除・整理が必要

### 初回実証で必要なのはどちらか
**初回はスクリーンショット方式で十分**である。
今回の目的は、まず「Figma差分 → ChatGPT整理 → Codex反映」の運用を回すことだからである。

## 10. ユーザーが実際にやるべき最短手順

ここだけ見れば初回作業を開始できる最短版を示す。

### 最短手順
1. ブラウザで Figma を開く
2. 新規デザインファイルを作る
3. 現行の Q-Scout 画面を Chrome で開く
4. Chrome DevTools の `Capture full size screenshot` でページ全体の画像を保存する
5. その画像を Figma にドラッグ＆ドロップする
6. 画像を複製して、片方を現状保管、片方を修正案用にする
7. 修正案用フレームの上に、テキスト・図形・矢印で変更内容を重ねる
8. 変更箇所に番号を付ける
9. 変更理由を短く書く
10. その Figma 差分を前提に ChatGPT へ「実装方針整理」を依頼する

## 14. 最終結論

初回の Figma 運用では、**ブラウザ版 Figma ＋ Chrome DevTools の full size screenshot 取得＋スクリーンショット取り込み**を標準手順とする。
これは最も失敗しにくく、すぐ始められ、今回の目的である「UI差分を視覚化して実装へ反映する」運用に十分だからである。

その後、必要になった場合のみ、Webページ取り込み系プラグインの利用を検討する。

以上を、Q-Scout for Spring におけるユーザー向け Figma運用作業手順書 v1.2 とする。
