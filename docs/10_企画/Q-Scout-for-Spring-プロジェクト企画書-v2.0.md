## 2. エグゼクティブサマリー

Q-Scout for Spring は、Spring Boot / Spring Framework を利用するプロジェクトに対し、設計健全性と Spring ベストプラクティス準拠度を可視化・定量評価するコード品質診断ツールである。

本プロジェクトは、従来の静的解析ツールのように警告を列挙するだけではなく、Spring 開発で生じやすい設計劣化や実装アンチパターンを、説明可能なルールベースで評価し、総合スコア・違反内容・改善出発点を Markdown レポートとして提示することを目的とする。

現行リポジトリでは、当初の CLI MVP 段階を超え、Spring Boot Web アプリケーションとしての利用導線、多言語化、日本語/英語の利用体験、preview / download 導線、サンプル評価運用を含む実用基盤が整備されている。

したがって、本企画書 v2.0 では、Q-Scout for Spring を「構想段階の品質診断案」ではなく、「Web / CLI 両対応の品質診断基盤」として再定義する。

## 3. 背景と課題認識

### 3.1 背景

Spring Boot / Spring Framework は、企業開発・業務システム・Web API 開発において広く利用される一方、コードレビュー品質の属人化、設計劣化の見えにくさ、Spring 流儀逸脱の早期検知困難といった課題が発生しやすい。

特に、以下のような問題はチーム開発において継続的にコストを生む。

- コードレビュー観点が担当者ごとにぶれる
- 技術負債や設計劣化が初期段階で可視化されにくい
- Spring の流儀から外れた実装を人手で継続監視する必要がある
- 品質判断をチームで共有しやすい形式に整理しにくい
- 改善優先度の判断基準が曖昧になりやすい

### 3.2 課題の本質

本プロジェクトが解決すべき本質課題は、単なる「Lint 警告不足」ではない。  
本質は以下である。

1. 設計品質評価の属人化
2. 技術負債の不可視化
3. Spring ベストプラクティス逸脱の見逃し
4. レビュー結果の説明責任不足
5. 改善優先度の定量化不足

## 4. プロジェクトの目的

Q-Scout for Spring の目的は、Spring プロジェクトに対して以下を実現することである。

1. 設計健全性を定量的に把握できる状態をつくる
2. Spring ベストプラクティス逸脱を早期に検知できる状態をつくる
3. 人間がレビューしやすいレポートを出力する
4. AI と接続しやすい入力形式を用意し、説明・改善提案へ接続しやすくする
5. Web / CLI の両利用導線を持つ品質診断基盤として成立させる

現行リポジトリ基準で、Q-Scout for Spring は以下の構成を持つ。

- Spring プロジェクト解析

## 7. 診断対象と評価観点

### 7.1 診断対象

- Spring Boot / Spring Framework プロジェクト
- Java ベースのコード資産
- Spring 設計品質、責務分離、実装流儀

### 7.2 現行の主要評価観点

現行の MVP では、以下の 6 ルールを中心に品質診断を行う。

1. Controller → Repository 直接参照
2. フィールドインジェクション
3. @Transactional の過不足候補
4. 例外握りつぶし
5. テスト欠落候補
6. パッケージ依存逆流候補

### 7.3 評価方式

- ルールベース評価
- 総合スコア表示
- 違反内容の説明可能な可視化
- 人間レビューと AI 接続を分離した出力

## 8. 差別化ポイント

Q-Scout for Spring の差別化は、単に「Spring 用の静的解析」であることではない。  
差別化の中核は以下にある。

### 8.1 Spring 設計品質への特化

汎用 Lint ではなく、Spring 開発現場で重要な責務分離、DI、Transaction、例外処理、テスト性、レイヤ構造を重視する。

### 8.2 スコアリング型の説明可能性

単純な警告一覧ではなく、品質状態をスコアとして把握でき、改善優先度を議論しやすい。

### 8.3 人間向け / AI 向け出力の明確分離

レビュー用 Markdown と AI 入力用 Markdown を分離しており、それぞれの目的に最適化しやすい。

### 8.4 AI 非依存コア

外部 AI API なしでも価値が成立する。  
AI は価値増幅要素であり、コア機能の前提条件ではない。

### 8.5 Web / CLI の両対応

ローカル検証、自己分析、将来的な公開利用を見据えた導線を同時に持つ。

## 12. リスクと対策

### リスク1：ルール精度に対する期待値ギャップ
対策：
- 候補検出重視であることを明示する
- ルールごとに改善フェーズを分離する
- 補助仕様で解釈基準を補う

### リスク2：文書と実装の乖離
対策：
- docs の正本 / 補助仕様 / 履歴資料の区分を維持する
- 企画文書も現行実装を踏まえて更新する
- README と docs/README.md を導線の基点とする

### リスク3：導入時の理解コスト
対策：
- Web / CLI の両方で利用可能にする
- サンプル出力と自己分析導線を活用する
- 人間向けレポートを中心に説明を行う

### リスク4：AI 依存と誤認されること
対策：
- AI 非依存コアであることを明記する
- AI は補助的価値増幅として位置づける
- AI 向け Markdown と人間向けレポートを分離する
