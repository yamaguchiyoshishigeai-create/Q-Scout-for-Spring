## 2. このプロジェクトの要約
本プロジェクトは、**Spring Framework / Spring Boot を利用した Java アプリケーションのコード品質・設計健全性・Springらしい実装度を静的解析し、スコアリングとレポート出力を行うツール「Q-Scout for Spring」** の企画・設計・実装を進めるものである。

出発点は CLI ベースの MVP であり、その後 **既存CLI資産を最大限流用しながらWebサービス化する方針** が確定している。さらに、Web利用を前提とした **多言語化対応** と、主要ルールの一つである **ControllerToRepositoryDirectAccessRule の見直し** が仕様対象として追加されている。

## 3. プロダクトの中核定義
### 3.1 目的
Q-Scout for Spring の目的は、属人的になりがちなコードレビューや設計品質評価を、Spring向けの静的解析とルール評価によって可視化・定量化することである。

### 3.2 主な提供価値
- 技術負債の可視化
- 品質のばらつき抑制
- レビューコスト削減
- Springベストプラクティス準拠度の可視化
- 開発者・テックリード・品質責任者・経営層に対する共通品質指標の提供

### 3.3 MVP時点の提供形態
- 当初は CLI ツール
- 解析対象は Spring Boot / Maven / 単一モジュール / Java 17 を前提
- 入力は対象プロジェクトパス、または Web化フェーズでは zip アップロード
- 出力は以下の3系統
  1. 標準出力サマリー
  2. 人間向け Markdown レポート
  3. AI投入用 Markdown

## 4. AI活用方針
本プロジェクトのAI利用は **段階的導入・非依存設計・コスト分離** が原則である。

### 4.1 フェーズ0の確定事項
- AI API は未使用
- AI投入用 Markdown を生成し、人間が ChatGPT / Gemini に手動投入する
- つまり **AIなしでも製品価値が成立する構造** を維持する

### 4.2 AIの役割
フェーズ0では AI は解析エンジンではなく、主に以下を担う。
- 違反理由の説明
- 改善提案の提示

## 5. CLI MVPの確定事項
### 5.1 基本アーキテクチャ
最小API設計では、以下の責務分離が確定している。
- cli
- application
- domain
- infrastructure

主要コンポーネントは以下。
- ProjectScanner
- RuleEngine
- ScoreCalculator
- ReportGenerator
- AiMarkdownGenerator

### 5.2 実行フロー
基本フローは以下。
1. Main.main(args)
2. CliApplication.run(args)
3. ArgumentParser が AnalysisRequest を生成
4. ProjectScanner.scan(request)
5. RuleEngine.analyze(projectContext)
6. ScoreCalculator.calculate(analysisResult)
7. ReportGenerator.generate(...)
8. AiMarkdownGenerator.generate(...)
9. ReportArtifact 生成
10. ExecutionSummary 生成
11. 標準出力へサマリー表示

### 5.3 スコアリング方式
- 100点満点の減点方式
- HIGH: -10
- MEDIUM: -5
- LOW: -2
- finalScore は 0 未満にしない

### 5.4 MVPで対象となる6ルール
1. ControllerToRepositoryDirectAccessRule
2. FieldInjectionRule
3. TransactionalMisuseRule
4. ExceptionSwallowingRule
5. MissingTestRule
6. PackageDependencyViolationRule

### 5.5 MVP実装方針
- 候補検出重視
- 誤検知ゼロは目標にしない
- 実装容易性と説明可能性を優先
- codeSnippet を可能な範囲で付与
- 人間向けレポートと AI向けレポートは分離
- 外部REST API はMVPでは対象外
- JSON出力もMVPでは対象外

## 6. 一次生成物ベースの現況認識
現況調査結果から、少なくとも以下の点は確認済みである。
- Main クラスが存在し、CLI として起動する構成である
- Spring Boot の Web 起動や REST API 起動は当初実装されていない
- jar 実行または classpath 指定での実行が可能な構成である
- 人間用 Markdown と AI用 Markdown の出力系がCLI前提で整備されている

したがって、**Web化はゼロからの再設計ではなく、CLI資産をラップして拡張するプロジェクト** と捉えること。

## 7. Web化フェーズの確定事項
### 7.1 基本方針
Web化フェーズでは、**CLI資産活用型Webラップ方式** が採用されている。

中核原則は以下。
- 解析コア不変
- 入口分離、コア共有
- 同期Web MVP
- Renderステートレス準拠
- Docker前提
- 過設計回避

### 7.2 Web MVP定義
以下が満たされれば Web MVP 完成とする。
1. ブラウザから zip をアップロードできる
2. 解析が実行される
3. スコアと結果が画面表示される
4. レポートをダウンロードできる
5. エラー時に適切なメッセージが出る

### 7.3 UI方針
- 単画面完結型アップロード実行MVP
- サーバレンダリング中心の最小構成
- 入力エリア、実行状態表示、結果表示、ダウンロード導線を持つ
- MVPでは複数画面・高度なSPA・ジョブ管理UIは採用しない

### 7.4 Webアーキテクチャ方針
- Spring Boot Web を採用
- CLI入口とWeb入口は分離
- 解析処理は SharedAnalysisService 等の共通サービスへ抽出
- Web層は薄く保つ
- 同期処理で完結させる
- 一時ファイルは /tmp 系ディレクトリ運用
- 成果物はファイルとして保存し、ダウンロード可能にする
- zip 解凍、ルート判定、クリーンアップを明示責務として扱う

### 7.5 Web実行制約
- 1リクエスト最大実行時間は60秒暫定
- MVPでは同期処理
- アップロード形式は zip のみ
- 最大ファイルサイズは20MB暫定
- 解凍後ルートに pom.xml が存在すること
- 不正形式は即時エラー
- 一時ディレクトリ運用を行う
- Render 前提のステートレス構成とする
- 将来は非同期ジョブ化を検討するが、MVPでは対象外

## 9. 多言語化対応の確定事項
多言語化対応仕様書では、以下が確定している。
- デフォルト言語は日本語
- 日本語 / 英語の切替を可能にする
- 主対象は Web UI 文言と人間用レポート文言
- AI用Markdown は原則英語維持
- Spring 標準の MessageSource ベース
- SessionLocaleResolver と LocaleChangeInterceptor を採用
- `?lang=ja` / `?lang=en` で切替可能
- 将来追加言語へ拡張しやすいキー設計にする

したがって、新規提案や実装検討では、**日本語中心UXを優先しつつ英語維持価値も損なわない構造** を前提にすること。

## 10. ControllerToRepositoryDirectAccessRule 見直しの確定事項
このルールは当初、Controller から Repository への直接参照候補を検出し、HIGH固定寄りの扱いで運用されていた。
しかし見直し方針書により、以下の方向へ更新する前提が整理されている。

### 10.1 見直し理由
- 直参照のすべてが同じ悪性度ではない
- 非許容ケースと条件付き許容ケースを分けたい
- 既存CLI/Web/レポート資産を壊さず精度と説明性を上げたい

### 10.2 新しい考え方
- まず直参照の存在を検出する
- その後、文脈に応じて重み付けする二段階方式を採る
- 書き込み系直参照など非許容ケースは HIGH
- 条件付き許容ケースは MEDIUM / LOW などに再分類する

したがって、このルールについて今後の議論や実装支援を行う際は、**一律断罪ではなく、重み付けと説明責務の強化** を前提とすること。
